# be-DailyJob

> 이 회고는 작업량을 보고하기 위한 문서가 아니라,
> 오늘 내가 어떤 판단을 했고 어떤 개발 행동을 반복하거나 바꿔야 하는지 확인하기 위한 기록입니다.
> 각 항목은 길게 쓰기보다, 사실/판단/근거/다음 행동이 드러나게 구체적으로 작성하세요.

### 1. 오늘 완료한 것

- 오늘 실제로 끝낸 기능/수정/학습:
  - 직무과제 댓글(Comment) 기능 7단계를 구현했다(7-1까지). `Comment` 모델(FK 역할의 `articleId` 추가), `CommentRepository` 인터페이스와 `InMemoryCommentRepository`(`ConcurrentHashMap` + `AtomicLong`), `CommentHandler`(POST 저장 + PRG 리다이렉트), `CommentFormHandler`(폼에 `articleId` hidden 주입, 템플릿 메서드 패턴)를 새로 만들고, `Router`, `AuthFilter`, `IndexHandler`, `index.html`, `comment/index.html`을 수정했다.
  - 순수 자바로 Spring MVC의 DispatcherServlet 파이프라인(RequestHandler → Parser → AuthFilter → Router → Handler → ResponseWriter)을 직접 만들어보며, 프레임워크가 대신 해주던 흐름을 내부 수준에서 이해했다.
  - 버그데이에 참여해 사소한 버그들을 발견, 수정했다(대표적으로 `<textarea>`의 `name=content` 누락으로 인한 조용한 실패).
  - 메인 프로젝트의 선행 주제를 실시간 경매 플랫폼으로 정했다. 이유는 단일 데이터에 대한 고밀도 동시성 제어(락킹)를 깊게 다뤄볼 수 있고, 기아 오토벨 같은 실제 경매 시스템에서 아이디어를 얻을 수 있어서다.

- (오늘의 목표 중에서) 아직 끝내지 못한 부분:
  - 직무과제를 7-1까지밖에 진행하지 못했다. 나머지 단계는 메인 프로젝트와 병행하거나 시간을 내 마무리해야 한다.

### 2. 오늘 가장 의미있었던 기술적 판단

- 내가 내린 판단:
  - 댓글 기능을 새로 설계하지 않고, 게시글(Article)이 이미 푼 `model → repository → handler → view` 4층 구조를 그대로 복제한 뒤 `articleId`(FK) 하나만 얹었다. 저장소는 인스턴스를 딱 하나만 만들어 저장 쪽(`CommentHandler`)과 읽는 쪽(`IndexHandler`)에 같은 참조를 주입했고, 작성자(`writerId`)는 폼이 아니라 세션에서 뽑았다.

- 그렇게 판단한 이유:
  - 댓글이 푸는 문제(작성 폼 → 저장 → 목록 조회 → 표시)가 게시글과 완전히 같은 모양이었다. 검증된 구조를 재사용하면 중복 검증 비용 없이 빠르고 안전하게 만들 수 있다. 저장소를 하나만 공유한 건, 인스턴스가 2개면 저장은 A의 store, 조회는 B의 store에서 일어나 방금 쓴 댓글이 영원히 안 보이기 때문이다. `writerId`를 세션에서 뽑은 건, 폼(hidden)은 조작 가능해 사칭 위험이 있는 반면 세션은 서버가 발급한 쿠키로만 조회되어 신뢰할 수 있기 때문이다(신원은 서버, 내용은 클라이언트).

- 고려했지만 선택하지 않은 대안:
  - 댓글용 구조를 처음부터 새로 설계하는 방식 → 이미 검증된 패턴을 두고 중복 설계하는 오버엔지니어링이라 배제했다.
  - 작성자를 폼에서 받는 방식 → 구현은 더 간단하지만 hidden 조작으로 `writerId=admin` 사칭이 가능해 신뢰 경계를 무너뜨리므로 배제했다.

- 이 판단이 코드 품질/테스트/유지보수에 미친 영향:
  - `PageHandler.handle()`을 `final`로 잠그고 `buildModel` 훅만 자식이 채우는 템플릿 메서드 패턴으로 중복을 제거하고 처리 순서를 강제했다. 리포지토리를 인터페이스에 의존하게 해(DIP) 나중에 `InMemoryCommentRepository`를 `JdbcCommentRepository`로 갈아끼워도 핸들러 코드는 그대로 두면 되고, 테스트 시 가짜(Fake) 구현으로 대체하기도 쉬워졌다.

### 3. 가장 막혔던 문제와 해결 과정

- 문제 상황 또는 증상:
  - `comment/index.html`에서 작성완료를 눌러도 댓글이 생기지 않고, 에러도 뜨지 않는 문제 발생했다.

- 처음 추측한 원인:
  - 저장 로직(`CommentHandler.save`)이나 저장소 공유(인스턴스가 2개로 갈렸나)가 문제일 것이라 의심했다.

- 원인을 확인하기 위해 시도한 방법:
  - 실제 전송되는 POST body와 `getParameter(content)`의 반환값을 확인했다.

- 실제 원인:
  - `<textarea>`에 `name=content`가 빠져 있었다. `name`이 없으면 브라우저가 그 입력을 POST body에 아예 싣지 않아, 전송되는 건 `articleId=5`뿐이었다. 그래서 `getParameter(content)`가 `null`이 되고 빈 내용 방어 분기로 빠져 저장 없이 리다이렉트됐다(잘못 복사된 `autocomplete=username`도 함께 있었다).

- 해결 방법:
  - `<textarea>`에 `name=content`를 추가하고 잘못 붙어 있던 `autocomplete` 속성을 제거했다.

- 다음에 비슷한 문제가 생기면 먼저 확인할 것:
  - 폼 전송 관련 버그는 서버 로직부터 의심하기 전에, 브라우저 개발자도구 Network 탭에서 실제로 전송된 요청 body를 먼저 확인한다. 각 입력의 `name`, `method`, `action`, `type=submit`이 제대로 있는지도 함께 본다.

### 4. 오늘 잘한 행동(개발과 관련된 것)

- 어떤 행동이었는지:
  - 새 기능(댓글)을 무작정 짜지 않고, 이미 검증된 게시글 구조에 1:N 차이(`articleId`) 하나만 매핑해 접근했다. 그리고 각 선택의 이유와 놓치기 쉬운 함정을 문서로 남겼다.

- 그 접근이 도움이 된 이유:
  - 구조가 일관되니 버그가 날 지점이 줄고, 나중에 코드를 다시 볼 때 왜 이렇게 했는지를 바로 설명할 수 있다.

- 다음에도 반복하고 싶은 점:
  - 새 문제를 만나면 이미 푼 비슷한 문제와 무엇이 같고 무엇이 다른지부터 짚고, 다른 점 하나에 집중하는 사고방식.

### 5. 오늘 아쉬웠던 행동(개발과 관련된 것)

- 비효율적이었던 접근/놓치거나 부족했던 점:
  - 직무과제를 7-1까지밖에 못 끝냈다. 또 그동안 사소한 버그를 사소하게 보며 기능 구현에 급급히 했던 순간들이 버그데이에서 실제로 문제로 드러나며 아쉬웠다. figma의 요구사항들 중 빼먹은 부분들이 몇개 씩 드러났다.

- 그렇게 된 원인:
  - 개념을 깊게 이해하고 문서로 정리하는 데 시간을 많이 쓰기도 했고, 어느 순간부터 진도가 늦춰지다 보니 핵심 기능 구현에 초점을 맞추다 보니, 버그 검증이나 테스트에 시간을 쏟지 못했던 것 같다.

- 다음에는 다르게 해볼 구체적인 행동:
  - 메인 프로젝트(실시간 경매 플랫폼)에서는 사소해 보이는 버그도 발견 즉시 고치고, TDD를 적극 도입해볼까 고민중이다. 솔직히 힘들다는 건 알지만, 충분히 고려해볼 예정이다. 내일은 기획 고도화를 위해 경매 도메인 레퍼런스, 케이스 스터디를 진행하고, Spring도 개인적으로 학습한다.
