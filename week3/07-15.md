# be-DailyJob (2026-07-16)

> 이 회고는 작업량을 보고하기 위한 문서가 아니라,
> 오늘 내가 어떤 판단을 했고 어떤 개발 행동을 반복하거나 바꿔야 하는지 확인하기 위한 기록입니다.
> 각 항목은 길게 쓰기보다, 사실/판단/근거/다음 행동이 드러나게 구체적으로 작성하세요.

### 1. 오늘 완료한 것

- 오늘 실제로 끝낸 기능/수정/학습:
    - 요청 파싱을 char 스트림에서 byte 스트림으로 전면 전환. `InputStreamReader`/`BufferedReader`로 소켓을 감싸던 걸 걷어내고, `HttpMessageReader`(내부 `BufferedInputStream`)를 새로 만들어 "HTTP 한 줄(`readLine`)"과 "정확히 N바이트(`readBytes` → `readNBytes`)"만 꺼내는 책임으로 분리했다.
    - 바디를 `byte[]`로 원본 보존. `HttpRequest`에 `byte[] body`를 방어적 복사(`body.clone()`)로 저장하고, form-urlencoded일 때만 `new String(body, UTF_8)`로 명시적 디코딩하도록 `HttpRequestParser.parseBody`를 정리했다.
    - 게시글 도메인/저장소 구현. `Article`(정적 팩토리 `create`, `like()`), `ArticleRepository` 인터페이스, `InMemoryArticleRepository`(`ConcurrentHashMap` + `AtomicLong sequence`로 AUTO_INCREMENT 흉내). `save`에서 `id == null`이면 `incrementAndGet()`으로 새 id 발급.
    - 글 작성 흐름 연결. `ArticleHandler`(POST 전용, 빈 내용이면 `/article`로 리다이렉트, 저장 후 `/`로), `Router`에 `/article/create` 등록, `AuthFilter`의 `LOGIN_REQUIRED`에 `/article`·`/article/create` 추가.
    - 템플릿 메서드 패턴 도입. `PageHandler.handle()`을 `final`로 고정(유저조회→헤더렌더→`buildModel`→200)하고, `buildModel`만 훅으로 열어 `IndexHandler`가 글 데이터(최신 글/`?id=`/이전·다음)를 채우도록 했다.
    - 개념 학습: `AtomicLong`/CAS/lost update, `volatile`과의 차이, DB `AUTO_INCREMENT`, JPA(인터페이스) vs Hibernate(구현체), `@GeneratedValue(IDENTITY)`, `JpaRepository` 파생 메서드/쿼리 메서드/`@Query`.
- (오늘의 목표 중에서) 아직 끝내지 못한 부분:
    - 글쓰기 Handler 리팩토링은 내일로 이월. 현재 `/write`, `/article`이 둘 다 그냥 `PageHandler`에 물려 있고, 작성 폼/생성 핸들러 쪽 구조 정리가 남았다.

### 2. 오늘 가장 의미있었던 기술적 판단

- 내가 내린 판단:
    - 요청 읽기를 문자(char) 스트림이 아니라 바이트(byte) 스트림으로 통일하기로 했다. 소켓을 `InputStreamReader`로 감싸지 않고 `BufferedInputStream`으로만 감싸, 헤더는 줄 단위로 읽되 바디는 `Content-Length`만큼 바이트 그대로 읽는다.
- 그렇게 판단한 이유:
    - HTTP `Content-Length`는 스펙상 바이트 수인데, 기존 코드는 `read(char[])`로 문자 수를 셌다. 영어는 1문자=1바이트라 우연히 맞았지만 한글은 UTF-8에서 3바이트라 길이가 어긋난다.
    - 더 결정적인 건 바이너리(이미지 업로드)다. `InputStreamReader`를 통과하는 순간 모든 바이트가 문자로 디코딩되는데, JPEG의 선두 `0xFF` 같은 바이트는 유효한 UTF-8이 아니라 `U+FFFD`로 치환되어 원본이 파괴된다. 앞으로 파일 업로드를 붙일 걸 생각하면 문자 변환 자체를 미뤄야 했다.
- 고려했지만 선택하지 않은 대안:
    - 문자로 읽되 UTF-8 인코딩 길이를 누적하는 방법: `read()`한 문자를 `getBytes(UTF_8).length`로 환산해 `bytesRead`에 더하며 `Content-Length`까지 채우는 방식. 한글 길이 문제는 풀리지만, 이미 `Reader`가 디코딩을 끝낸 뒤라 바이너리는 여전히 못 다룬다. 증상(한글 길이)만 막는 땜빵이라 버렸다.
- 이 판단이 코드 품질/테스트/유지보수에 미친 영향:
    - "읽기(바이트 꺼내기)"는 `HttpMessageReader`, "조립"은 `HttpRequestParser`, "보관"은 `HttpRequest`로 책임이 한 곳에 하나씩 모였다.
    - 텍스트 해석을 필요한 지점에서만 명시적 charset으로 하게 되어, 인코딩 관련 버그가 날 곳이 좁아졌다. `readNBytes`가 TCP 분할 수신을 알아서 반복 처리해줘서 부분 읽기 루프도 사라졌다.

### 3. 가장 막혔던 문제와 해결 과정

- 문제 상황 또는 증상:
    - POST 바디에 한글이 들어오면 읽어들인 바디 길이가 어긋났다. 이후 이미지 업로드까지 고려하니 지금 구조로는 아예 안 된다는 판단.
- 처음 추측한 원인:
    - "`Content-Length`(바이트)와 내가 읽는 문자 수가 안 맞아서" → 문자를 UTF-8로 재인코딩해 바이트를 세면 되겠다고 생각했다.
- 원인을 확인하기 위해 시도한 방법:
    - 스트림 파이프라인을 다시 짚었다. `new BufferedReader(new InputStreamReader(in))`을 만드는 그 순간부터 이미 바이트→문자 디코딩이 일어난다는 점, `BufferedReader`는 줄 읽기 편의일 뿐 디코딩 주체가 아니라는 점을 확인했다.
- 실제 원인:
    - 길이 불일치는 표면 증상이고, 근본 원인은 읽는 계층에서 너무 일찍 문자로 디코딩한 것. 그래서 바이너리 무손실 보존이 원천적으로 불가능했다.
- 해결 방법:
    - 소켓 → `BufferedInputStream`(버퍼링만, 디코딩 X) 구조로 바꾸고, `RequestHandler`가 `Reader`로 감싸지 않은 `InputStream in`을 그대로 `parse(in)`에 넘기게 했다. 헤더는 `readLine`으로 줄 단위, 바디는 `readNBytes(Content-Length)`로 바이트 단위. `\r`은 버리고 `\n`을 줄 종료로 삼는다.
- 다음에 비슷한 문제가 생기면 먼저 확인할 것:
    - "어느 계층에서 인코딩/디코딩이 일어나는가"를 가장 먼저 그린다. 길이·깨짐 증상이 보이면 값을 고치기 전에 "바이트로 다뤄야 하는 구간을 문자로 바꾸고 있지 않은지" 확인한다.

### 4. 오늘 잘한 행동(개발과 관련된 것)

- 어떤 행동이었는지:
    - 한글 길이만 막는 임시방편에서 멈추지 않고, 파이프라인 근본(문자 vs 바이트)을 바꾸는 결정까지 갔다. 그리고 그 변경을 `byte 단위로 읽기` → `byte[] 보관` → `Reader 제거` → `HttpMessageReader 도입`처럼 작은 커밋으로 쪼개서 진행했다.
- 그 접근이 도움이 된 이유:
    - 증상 대신 원인을 고쳐서 앞으로의 파일 업로드 요구사항을 미리 흡수했다. 커밋을 쪼갠 덕에 각 단계가 무엇을 바꿨는지 히스토리로 남아, 되돌리거나 면접에서 설명하기 쉽다.
- 다음에도 반복하고 싶은 점:
    - "이 수정이 증상 치료인가 원인 치료인가"를 스스로 묻는 것. 그리고 리팩토링을 의미 단위 커밋으로 분리하는 습관.

### 5. 오늘 아쉬웠던 행동(개발과 관련된 것)

- 비효율적이었던 접근/놓치거나 부족했던 점:
    - 처음부터 char 스트림으로 설계했던 것에 대한 아쉬웠다. 바이너리 가능성을 초기에 안 따져서 재작업이 생겼다.
    - 오늘 만든 파싱/저장소 로직에 테스트를 안 붙였다. 특히 한글 바디 길이, form-urlencoded 파싱, `InMemoryArticleRepository`의 `save`/`findLatest`/`findPrevious`/`findNext` 경계(글 0개, id 최소/최대)는 검증 없이 넘어갔다.
    - `AtomicLong`은 `incrementAndGet`으로 잘 썼지만, "`get` 후 `set`은 원자적이지 않다"는 함정을 학습만 하고 실제 코드에서 그런 패턴이 없는지 훑어보진 못했다.
- 그렇게 된 원인:
    - 기능 연결(글 작성 → 화면 표시)을 눈으로 확인하는 데 집중해서, 자동 검증을 뒤로 미뤘다.
- 다음에는 다르게 해볼 구체적인 행동:
    - 내일 글쓰기 Handler 리팩토링을 하기 전에 `InMemoryArticleRepository` 경계 테스트와 한글 바디 파싱 테스트부터 작성한다(리팩토링 안전망 확보).
    - 새 파싱/저장 로직을 추가할 때는 "정상 1개 + 경계(빈/최대) 1개" 최소 테스트를 같이 커밋하는 규칙을 시도한다.
