  - 학습: `instanceof` 패턴 매칭(캐스팅 자동화), Javadoc(주석→AST→HTML 생성 흐름과 `@param`/`@throws`/`{@inheritDoc}`), CRUD↔HTTP 메서드 매핑, 안전성/멱등성, RESTful URL 설계 원칙, checked vs unchecked 예외 판별법.
  - 리팩터링(구현): `methodNotAllowed(String)` → `methodNotAllowed(HttpMethod...)` 타입 안전화.
  - 리팩터링(구현): `HttpResponse` 팩토리 메서드 6개에 흩어져 있던 body/headers/Content-Length 조립 로직을 빌더 패턴으로 통합.
  - 리팩터링(구현): 각 핸들러가 `return`으로 직접 응답을 만들던 방식을 → `throw` + `ExceptionResolver` 중앙 처리 구조로 전환. 500 에러도 이제 에러 페이지로 응답.

### 2. 오늘 가장 의미있었던 기술적 판단

- 내가 내린 판단:
  - `HttpResponse`의 팩토리 메서드(`ok`, `found`, `notFound`, `methodNotAllowed`...)에 중복되던 조립 로직을 빌더 패턴으로 통합했다. `status(HttpStatus)`를 시작점으로 두고, `header()`/`body()`는 `return this`로 체이닝, `build()`에서 공통 로직을 처리하는 구조.
- 그렇게 판단한 이유:
  - 6개 메서드에 `headers.put("Content-Length", ...)`가 똑같이 반복되고 있었다. 새 응답을 추가할 때마다 Content-Length를 깜빡할 위험이 있었다. 빌더의 `build()` 한 곳에서 `body.length`로 자동 계산하게 하면 이 실수가 원천 차단된다. 이게 리팩터링의 실질적 이득.
  - 필수값(status)은 빌더 시작 메서드의 인자로 강제하고, 선택값(header/body)은 세터로 둬서 "status 없이 build()" 같은 실수를 컴파일 단계에서 막을 수 있다.
- 고려했지만 선택하지 않은 대안:
  - 팩토리 메서드 유지 + 중복 감수: 지금 당장은 동작하지만 응답 종류가 늘수록 중복과 누락 위험이 커진다.
  - record 생성자 직접 노출: 필수/선택값 구분이 안 되고, 호출부가 매번 `Map`과 `Content-Length`를 직접 만들어야 해서 실수가 호출부로 분산된다.
- 이 판단이 코드 품질/테스트/유지보수에 미친 영향:
  - 팩토리 메서드는 그대로 유지되어 호출부는 무변경(`HttpResponse.found("/")` 그대로), 내부 조립만 빌더로 위임. Spring `ResponseEntity`가 쓰는 "팩토리 + 빌더 2층 구조"와 같은 형태로, 공통 규칙을 한 곳에 모아 유지보수 지점을 단일화했다.

### 3. 가장 막혔던 문제와 해결 과정

- 문제 상황 또는 증상:
  - 어떤 메서드는 `try/catch`로 예외를 잡거나 `throws`를 달아야 컴파일이 되고, 어떤 건 안 그래도 됐다. "이게 checked라서 throws가 필요한지"를 미리 어떻게 판단하는지가 헷갈렸다.
- 처음 추측한 원인:
  - 예외 이름을 그냥 외워야 하나 싶었다(IOException, SQLException은 처리, NullPointer는 안 함...).
- 원인을 확인하기 위해 시도한 방법:
  - 컴파일러 에러 메시지 확인, IDE 빨간 줄, Javadoc의 Throws 섹션 확인, 그리고 cmd+클릭으로 메서드 시그니처의 `throws`절 → 그 예외의 상속 체인을 따라가 봄.
- 실제 원인:
  - checked/unchecked는 상속 체인으로 결정된다. `RuntimeException`을 상속하면 unchecked(throws 불필요), `Exception`을 직접 상속하면 checked(throws 또는 catch 강제). 예: `read()`가 `throws IOException` → `IOException extends Exception` → checked.
- 해결 방법:
  - "시그니처의 `throws`절을 먼저 보고 → 그 예외가 `RuntimeException` 계열인지 확인"하는 순서로 판단. 낯선 API는 외우지 말고 시그니처부터 타고 들어간다.
- 다음에 비슷한 문제가 생기면 먼저 확인할 것:
  - 새 API 호출 시 메서드 시그니처의 `throws`절부터 확인. 컴파일러/IDE가 강제하는 건 checked라는 신호로 받아들이기.

### 4. 오늘 잘한 행동(개발과 관련된 것)

- 어떤 행동이었는지:
  - `methodNotAllowed`의 파라미터를 String으로 두려다 "이 타입이 최선인가? `getMethod()`는 이미 `HttpMethod` enum을 쓰는데 왜 여기만 String이지?"를 스스로 질문했다. 그리고 선택지(String / 단일 enum / varargs / Set)를 표로 비교하고 채택 근거를 문서로 남겼다.
- 그 접근이 도움이 된 이유:
  - "동작만 되는 코드"에서 멈추지 않고, 타입 안정성(오타를 런타임→컴파일 타임으로 당김), 도메인 모델 일관성(primitive obsession 회피), 직렬화 경계 개념까지 근거가 정리됐다. Allow 헤더가 다중 메서드라 varargs가 맞다는 결론도 근거에서 자연스럽게 나왔다.
- 다음에도 반복하고 싶은 점:
  - 결정마다 대안과 트레이드오프를 표/문서로 남기기. 나중에 "왜 이렇게 했지"를 재현할 수 있고 면접 대비도 된다.

### 5. 오늘 아쉬웠던 행동(개발과 관련된 것)

- 비효율적이었던 접근/놓치거나 부족했던 점:
  - H2, JDBC, JPA, instanceof, Javadoc, REST, 예외처리, 빌더까지 개념 정리에 시간을 크게 쓰면서, 정작 PR1의 실제 산출물(H2/JDBC로 Repository 교체)은 진도를 못 뺐다.
- 그렇게 된 원인:
  - "개념을 완벽히 이해한 뒤 구현하자"는 성향. 계획(PR1: 인메모리로 먼저 관통 → 이후 H2 교체)이 있었는데도 이론을 앞세웠다.
- 다음에는 다르게 해볼 구체적인 행동:
  - 계획대로 인메모리 구현으로 먼저 세로 관통(모델→Repository→핸들러→렌더)을 끝내 동작하는 상태를 만든 뒤, H2 구현체로 교체한다. 개념은 구현하다 막히는 지점에서 필요한 만큼만 판다.
