> 오늘 내가 어떤 판단을 했고 어떤 개발 행동을 반복하거나 바꿔야 하는지 확인하기 위한 기록입니다. 
> 각 항목은 길게 쓰기보다, 사실/판단/근거/다음 행동이 드러나게 구체적으로 작성하세요.

### 1. 오늘 완료한 것
- 오늘 실제로 끝낸 기능/수정/학습:
  - [Entity 매핑 회의]
    - 엔티티를 연관관계(@ManyToOne)로 갈지 id 참조로 갈지 팀이랑 회의하면서 여러 방향을 논의함. 큰 줄기는 이랬음.
      - Bid에 JPA 연관관계를 도입할지: Bid는 User랑 Auction을 물고 있는데, 읽기는 fetch join으로 N+1을 커버할 수 있지만 쓰기는 쿼리 개수가 달라지는 문제라 연관관계로 해결이 안 됨.
      - ManyToOne이냐 id냐 나누는 기준: bid나 auction처럼 생명주기가 다르거나 같아도 개수가 많은 건 id로 분리하고, vehicle처럼 생명주기가 같은 건 cascade로 처리하는 쪽으로 기준을 잡음.
      - 애그리거트는 패키지 단위로 묶고 다른 패키지끼리는 독립된 애그리거트로 봄. MSA로 쪼갤 때 유리하고 확장성에도 좋음.
      - 양방향은 안 쓰기로 함. 학습할 땐 서로 객체를 물고 있으면 편하지만, 실무에선 양쪽 다 갱신해줘야 해서 영속성 보장 문제랑 순환참조가 생김. 우리는 다 단방향으로만 감.
      - fetch join 단점도 짚음. 페이징이 DB가 아니라 애플리케이션 메모리에서 처리돼서 카티션 프로덕트로 메모리가 커지고, 핫패스에서 과다조회로 I/O랑 메모리가 낭비되고, 쿼리 재사용성이 떨어짐. 재사용은 QueryDSL로 필요한 메서드만 뽑아 쓰면 되는데, 이거 하나 때문에 QueryDSL을 넣는 게 맞나는 얘기가 남음.
      - Setter를 안 쓰는 이유는 JPA 엔티티에서 객체 안정성 때문에 @Setter를 최소화하는 게 관례라서. DTO를 두는 건 DDD로 가면서 도메인별로 분리하려는 거고.
    - 결론은 전면 id 참조는 오버엔지니어링(JPA 편의를 통째로 버리고 지금은 확장 이점도 실체가 없음), 전면 연관관계는 편하지만 N+1 함정이라 가벼운 하이브리드로 감. 기본은 연관관계 + 전부 LAZY(이미 적용됨), 조회는 fetch join/@EntityGraph로 N+1 차단 의무화, 양방향은 꼭 필요할 때만, Bid만 id 참조(auctionId/bidderId — 쓰기 몰림 + 경계 밖 + 조인 불필요 3조건 다 맞는 유일한 엔티티), Notification.referenceId는 다형 참조라 id 유지, FK 제약은 DB 레벨에서 유지. 다만 User를 가리키는 참조들(Vehicle.seller, Deal.seller/buyer, Auction.winner, Evaluation.evaluator, Notification.user)을 전부 Long userId로 통일할지는 결론을 못 냈음.
  - [PR 코멘트로 팀원과 토론]
    - 오늘 처음으로 PR에 코멘트를 남기면서 팀원이랑 토론해봄. 세 군데 남겼음.
      - BigDecimal(precision=15, scale=2)로 잡힌 가격 컬럼: 원화는 소수점을 안 쓰는데 저렇게 하면 1000000.00처럼 항상 .00이 붙어서, 국내 중고차 경매면 scale=0이나 long이 낫겠다고 함. 작성자가 long으로 바꿨고, int는 약 21억 한계라 고가 차량에서 오버플로우 위험이 있어서 Auction.startPrice/currentPrice, Vehicle.estimatedPrice, Bid.amount 전부 long으로 통일함.
      - plateNumber에 unique 제약이 없는 게 의도인지 물어봄. 작성자는 의도적으로 안 걸었다고 함. 중고차는 같은 차가 세월 지나 재판매로 다시 나오는 게 정상이라 전역 unique를 걸면 재등록을 막아버리고, 진짜 식별자는 번호판보다 VIN에 가깝고 번호판은 이전등록으로 바뀌기도 함. 정말 막아야 할 건 동시에 활성 매물 2개인데 그건 컬럼 unique로 안 되고 애플리케이션 레벨로 푼다고 함.
      - Bid처럼 불변 레코드에 BaseTimeEntity를 상속하면 updated_at이 생기는데 의미 없는 필드 아니냐고 물어봄. BaseCreatedEntity/BaseTimeEntity 2단 분리도 가능하지만, 완전 불변인 게 Bid 하나뿐이라(나머지는 status·isRead로 상태가 바뀜) 하나 때문에 상위 클래스를 나누는 건 오버라 판단해서 통일 유지하고 ERD를 코드에 맞춰 갱신하기로 함.
  - [개인 공부 - JPQL / fetch join]
    - JPQL은 DB 테이블이 아니라 엔티티 객체를 대상으로 조회하는 쿼리 언어라는 걸 정리함. SQL이 테이블/컬럼을 대상으로 한다면 JPQL은 자바 엔티티/필드를 대상으로 하고 대소문자를 엄격히 구분함. JPA가 객체 중심으로 코드를 짜니까 테이블이 아니라 객체 기준으로 쿼리를 날리려고 존재하고, 하이버네이트가 현재 DB 방언에 맞춰 SQL로 번역해줘서 MySQL을 Postgre나 Oracle로 바꿔도 JPQL을 안 고쳐도 됨.
    - fetch join은 JPA 전용 문법으로, 일반 JPQL join이 연관 엔티티를 프록시로 남겨두는 것과 달리 조회 시점에 INNER/LEFT JOIN으로 연관 객체 그래프까지 한 방에 끌어와 영속성 컨텍스트에 진짜 객체로 채워둠. 그래서 member.getTeam().getName()을 불러도 추가 SQL이 안 나가서 N+1이 해결됨.
  - [개인 공부 - Foreign Key 매핑 방식]
    - JPA에서 FK를 다루는 두 방식을 정리함. 하나는 @ManyToOne + @JoinColumn으로 상대 엔티티 객체를 통째로 물고 있는 표준 방식(FK 제약이 자동 생성됨, order.getUser()로 타고 들어가기 편함), 다른 하나는 @Column으로 Long userId만 저장하는 방식(JPA는 FK인 줄 모름, 조회는 Repository로 직접 다시 해야 함, MSA나 논리적 FK 환경에서 자주 씀).
    - FK 제약 부여 위치도 정리함. DB 단은 @ForeignKey로 실제 제약 DDL을 만들어 엔진이 무결성을 100% 보장하지만 Lock 때문에 쓰기가 상대적으로 느리고 대용량 이전·삭제가 번거로움(금융/결제/초기 서비스에 적합). 애플리케이션 단은 ConstraintMode.NO_CONSTRAINT로 DB 제약을 억제하고 서비스 레이어에서 존재 검증을 직접 하는 방식이라 Lock 경합이 적어 쓰기가 빠르고 DDL 변경·이관이 자유로움(C2C/커머스, 대용량, MSA에 적합).
  - [개인 공부 - LAZY / 프록시 / N+1]
    - LAZY가 실제로 뭔지 프록시 관점으로 정리함. id 참조든 연관관계든 Bid를 조회하면 auction 자리엔 가짜 프록시 껍데기만 들어가 있다가 bid.getAuction().getStartPrice()를 부르는 순간 그제서야 추가 SQL이 나감(LAZY). EAGER는 조회 때 무조건 같이 가져옴.
    - Bid를 id 참조로 빼는 진짜 이유도 다시 정리함. 저장(쓰기) 자체는 getReference로 프록시만 챙기면 SELECT가 안 나가서 연관관계든 id든 차이가 없음. 진짜 차이는 읽기에서 나는데, 연관관계로 두면 루프에서 무심코 getAuction/getBidder를 부르면 N+1이 터지는 footgun이 있고, id로 두면 그냥 숫자라 실수로 터질 일이 없는 대신 필요하면 WHERE id IN (...)으로 내가 명시적으로 가져와야 함. 그래서 Bid를 id로 빼는 근거는 쓰기 성능이 아니라 읽기 N+1 footgun 제거 + 애그리거트 경계임.
  - [AWS 강의 1주차]
    - VPC 기본 구조를 정리함. VPC에 IGW를 달아 구멍을 열고, Route table은 VPC에 기본 1개에 subnet별로 보통 1개씩 더 붙이고, EC2는 Security Group으로 막혀 있어서 쓰려면 특정 포트를 열어야 함.
    - EC2는 AWS가 주는 VM이고 stop(멈춤)과 terminate(아예 끔)가 있는데 terminate는 하지 말라고 함. 인스턴스 타입은 t4g.micro면 T타입 4세대 graviton 프로세서 micro 크기 식으로 읽고, 사이즈가 small에서 large로 가면 CPU/메모리가 2배씩 늘어남.
    - 스토리지는 S3랑 EBS를 비교함. S3는 오브젝트 단위에 용량 무제한이고 HTTP REST로 어디서든 독립 접근되며 정적 파일 호스팅·프론트 배포·백업에 씀. EBS는 블록 단위에 볼륨당 최대 64TB, EC2에 붙여야 로컬 디스크처럼 쓰고 OS/부트 볼륨·DB·빠른 I/O에 씀. 대량 정적 파일이나 여러 서비스 동시 접근은 S3, 빠른 I/O나 부분 수정은 EBS.
    - IAM도 정리함. User/Group/Role은 인증, Policy는 인가. User는 개인 정책을 갖고 Group은 정책을 상속시켜 여러 명에게 동시에 주고, Role은 두 성격이 있는데 User가 쓰면 기존 권한을 덮어씌우고(반납하면 원래 권한 복귀) AWS 리소스가 쓰면 EC2가 S3에 접근하게 IAM 방화벽을 뚫어줌. Policy는 ALLOW/DENY를 담은 json 문서.
    - 그 외 서버리스도 훑음. Lambda는 서버가 없는 게 아니라 서버 관리 부담이 사라진 거고 트래픽 있을 때만 코드가 실행됨. 최대 실행 15분 제한, cold start(주기 호출이나 concurrency로 회피), stateless라 상태는 S3 같은 외부 저장소에 둬야 함, 메모리만 올려도 CPU/네트워크가 같이 오르고, 실행 역할(IAM Role)과 트리거가 필요함. DynamoDB는 서버리스 NoSQL로 파티션 키가 분산을 결정하고 일관성은 기본 최종일관성 + 강력한 일관성 선택, CAP에서 P는 필수고 CP/AP를 고름. S3는 버킷(리전 종속, 전역 고유 이름)에 오브젝트를 담고 키로 식별(경로처럼 보여도 폴더 아님), 기본 Private에 presigned URL로 임시 접근을 줌.
- (오늘의 목표 중에서) 아직 끝내지 못한 부분:
  - User를 가리키는 참조들(Vehicle.seller, Deal.seller/buyer, Auction.winner, Evaluation.evaluator, Notification.user)을 전부 Long userId로 통일할지 결론을 못 냄. 성능 문제가 아니라 DDD 경계를 얼마나 엄격히 그을지의 컨벤션 문제라, Bid만 id 참조로 바꾸고 나머지는 연관관계 편의를 챙기는 쪽으로 가려다가, 어 이럴 거면 이거 결국 그냥 전면 JPA(연관관계) 하는 거랑 뭐가 다르냐는 지점에서 막혀서 정하지 못하고 끝남.

### 2. 오늘 가장 의미있었던 기술적 판단
- 내가 내린 판단:
  - 엔티티를 전면 id 참조도 전면 연관관계도 아닌 가벼운 하이브리드로 가고, id 참조는 Bid 하나에만 적용하기로 함.
- 그렇게 판단한 이유:
  - 전면 id 참조는 JPA 편의를 통째로 버리는 건데 지금은 확장 이점이 실체가 없어서 오버엔지니어링이고, 전면 연관관계는 편하지만 N+1 함정이 큼. Bid는 쓰기가 몰리고 애그리거트 경계 밖이고 조인이 불필요한 3조건을 다 만족하는 유일한 엔티티라, 여기만 id로 빼면 읽기 N+1 footgun을 없애면서 나머지는 연관관계의 읽기 편의를 그대로 챙길 수 있음.
- 고려했지만 선택하지 않은 대안:
  - 전면 id 참조. 경계는 확실히 그어지지만 JPA 편의를 다 버리고 조회할 때마다 Repository로 다시 가져와야 해서 지금 규모엔 과함. 전면 연관관계도 편하긴 한데 어디서든 N+1이 터질 수 있어 배제함.
- 이 판단이 코드 품질/테스트/유지보수에 미친 영향:
  - 대부분 연관관계 + LAZY로 통일되면서 조회 코드가 단순해지고, 조회는 fetch join/@EntityGraph로 N+1을 막는 규칙을 팀 컨벤션으로 세울 수 있게 됨. Bid만 예외로 두면서 예외의 근거(쓰기 몰림·경계·조인 불필요)도 명확해짐. 다만 User 참조 통일 여부가 미결이라 컨벤션에 구멍이 하나 남음.

### 3. 가장 막혔던 문제와 해결 과정
- 문제 상황 또는 증상:
  - 엔티티를 연관관계로 둘지 id 참조로 둘지, 특히 Bid를 id로 빼는 근거를 팀에서 쓰기가 몰려서라고 들었는데 그 설명이 계속 애매하게 느껴짐.
- 처음 추측한 원인:
  - 입찰을 저장할 때 연관관계면 프록시를 챙기는 비용이 들어서 쓰기가 느려지는 거라고 생각함.
- 원인을 확인하기 위해 시도한 방법:
  - 쓰기와 읽기를 쪼개서 각각 어떤 SQL이 나가는지 따져봄. 저장할 때 getReference로 껍데기만 챙기면 SELECT가 안 나가는 것도 확인함.
- 실제 원인:
  - 쓰기 자체는 연관관계든 id든 INSERT 한 방이라 차이가 없었음. getReference가 DB를 안 치기 때문에 프록시 챙기는 비용이라는 말은 사실상 틀림. 진짜 차이는 읽기에서 나는데, 연관관계로 두면 루프에서 무심코 getAuction/getBidder를 불러 N+1이 터지는 footgun이 있는 거였음.
- 해결 방법:
  - Bid를 id로 빼는 근거를 쓰기 성능이 아니라 읽기 N+1 footgun 제거 + 애그리거트 경계로 다시 잡음. 결론(Bid를 id로)은 그대로지만 근거가 정확해짐.
- 다음에 비슷한 문제가 생기면 먼저 확인할 것:
  - 성능이 이유라고 들으면 쓰기랑 읽기를 쪼개서 실제로 어느 쪽에서 무슨 쿼리가 나가는지 SQL 단위로 따져볼 것. 근거가 틀려도 결론은 맞을 수 있으니 결론만 받지 말고 근거를 검증할 것.

### 4. 오늘 잘한 행동(개발과 관련된 것)
- 어떤 행동이었는지:
  - 처음으로 PR에 코멘트를 남기면서 팀원과 토론함. 가격 컬럼 타입, plateNumber unique, 불변 레코드의 updated_at 세 군데를 근거를 붙여 질문하고 작성자 답을 받아 결론까지 냄.
- 그 접근이 도움이 된 이유:
  - 그냥 넘어갔으면 .00이 붙는 컬럼이나 오버플로우 위험이 남았을 텐데, 질문으로 long 통일까지 실제 수정을 이끌어냄. unique나 updated_at처럼 내가 틀렸을 수 있는 것도 물어보니 작성자의 의도(재판매 시나리오, 불변 엔티티가 하나뿐)를 알게 돼서 서로 설계 맥락을 맞출 수 있었음.
- 다음에도 반복하고 싶은 점:
  - 지적이 아니라 의도를 묻는 방식으로 코멘트를 남기는 것. 내가 맞다고 단정하지 않고 왜 이렇게 했는지 물으면 토론이 됨.

### 5. 오늘 아쉬웠던 행동(개발과 관련된 것)
- 비효율적이었던 접근/놓치거나 부족했던 점:
  - 엔티티 매핑 논의를 오래 했는데 User 참조를 전부 Long userId로 통일할지는 결론을 못 내고 끝남. Bid만 id 참조로 바꾸고 나머지는 연관관계 편의를 챙기자는 쪽으로 가려다가, 어 이거 그럼 그냥 전면 JPA 하는 거랑 뭐가 다르냐는 데서 막혀버림.
- 그렇게 된 원인:
  - 성능 문제가 아니라 DDD 경계를 어디까지 엄격히 그을지의 컨벤션 문제라 정답이 없고, 판단 기준(경계 엄격함 vs 읽기 편의)을 먼저 합의하지 않은 채 엔티티별로 하나씩 얘기하다 보니, 최소 변경으로 가면 결국 전면 연관관계랑 차이가 없어지는 게 걸려서 계속 막혔음.
- 다음에는 다르게 해볼 구체적인 행동:
  - 컨벤션 성격의 결정은 엔티티를 하나씩 보기 전에 판단 기준부터 팀이 합의하고(예: 경계를 엄격히 그을지 읽기 편의를 우선할지), 그 기준으로 나머지를 일괄 적용하기. 결론 못 낸 User 참조 통일 건은 다음 회의 안건으로 올려두기.
