> 오늘 내가 어떤 판단을 했고 어떤 개발 행동을 반복하거나 바꿔야 하는지 확인하기 위한 기록입니다. 
> 각 항목은 길게 쓰기보다, 사실/판단/근거/다음 행동이 드러나게 구체적으로 작성하세요.

### 1. 오늘 완료한 것
- 오늘 실제로 끝낸 기능/수정/학습:
  - [역할 분배 확정]
    - 아침에 MVP 단위로 트랙을 4개로 쪼개고 각자 뭘 하고 싶은지 구체화해서 완만하게 배정을 끝냄. 나는 B(입찰, 동시성)를 맡기로 함.
      - A 경매 코어: 경매 게시글 작성/발행, 경매방 자동 개설 스케줄링(30분 전), 상태머신(대기 -> 진행중 -> 종료/유찰), 소프트 클로즈 엔진과 연장 횟수 기록, 마감 스케줄러, 종료/낙찰 확정/유찰 처리, 서버 시각 기준 판정의 단일 소스 제공(전 트랙이 의존). 최고 난이도.
      - B 입찰, 동시성(내 트랙, A와 페어): 입찰 접수 API와 검증 파이프라인, 구간별 최저 상승가 정책(테이블화, 하드코딩 금지), 비관적 락/원자적 처리, 입찰 이력 저장과 커서 기반 페이지네이션, 소프트 클로즈 트리거는 A의 도메인 서비스 호출로 위임(경계 명확히), 동시성 테스트 코드 작성 책임.
      - C 실시간 통신, 알림: WebSocket/SSE 세션 관리와 입퇴장, 호가창 브로드캐스트, 참여자 수와 입찰자 수 분리 산출, 닉네임 마스킹, 알림 도메인 전체, 외부 입찰(딜러) 유입분 지표 반영.
      - D 차량, 회원, 거래: 회원/인증과 권한, 차량 등록과 이미지 업로드(외부 스토리지), 외부 시세 API 연동, 평가사 흐름과 평가 완료 차량 시드 데이터, 낙찰 후 거래 상태머신(CONFIRMED -> IN_TRANSIT -> COMPLETED), 탁송/배송 정보 입력.
    - 소프트 클로즈처럼 트랙이 물리는 지점은 A가 도메인 서비스로 소스를 쥐고 B가 트리거만 위임 호출하는 식으로 경계를 미리 그어둠.
  - [GitHub Project 칸반보드 도입]
    - Project를 처음 적용해서 칸반보드 작성을 끝냄. 앞으로 issue를 파면 자동으로 보드에 올라가 Todo에서 In Progress, Done으로 흐르게 하고, Todo에 백로그를 쌓아두는 형태로 운영하기로 함.
  - [Entity 매핑, JPA 연관관계로 최종 확정]
    - 어제 못 정한 참조 방식을 오늘 JPA 객체 연관관계로 확정함. 근거는 아래대로 정리해서 남김.
      - 단일 모놀리식이고 경매 상세나 입찰 내역 화면에서 차량, 게시글, 입찰자를 함께 조회하는 경우가 많음. 입찰 내역에선 금액뿐 아니라 입찰자 닉네임도 자주 필요해서, Bid에서 User를 연관관계로 매핑하면 JPQL이나 QueryDSL로 자연스럽게 조인할 수 있음.
      - ID 참조는 애그리거트 경계를 엄격히 분리하거나 서비스 간 결합을 줄일 때 이점이 크지만, 지금은 경계를 엄격히 나누거나 MSA로 확장할 필요가 명확하지 않음. 처음부터 전부 ID로만 관리하면 조회 코드와 Repository 호출이 오히려 복잡해짐.
    - 연관관계 단점을 줄이기 위한 규칙도 같이 확정: 모든 ManyToOne과 OneToOne은 LAZY, 양방향 최소화(단방향 기본), Auction의 List<Bid>처럼 계속 커지는 컬렉션은 매핑 안 하고 Repository 페이징, 상세는 fetch join이나 EntityGraph, 목록은 DTO Projection, Cascade는 차량과 차량 이미지처럼 생명주기가 같은 관계에서만.
    - N+1 정리: 연관관계라고 무조건 터지는 게 아니라 LAZY를 반복문에서 접근할 때 문제. API별로 필요한 연관관계를 명확히 정의하고 상세는 fetch join, 목록은 DTO Projection으로 처리. 엔티티를 API 응답으로 직접 반환하지 않아 직렬화 중 예상치 못한 지연 로딩이 없게 함.
    - 결론: ID 참조를 몰라서 안 쓴 게 아니라, 지금 프로젝트에선 객체 연관관계의 조회 편의와 유지보수성이 더 크다고 판단해 선택. 향후 서비스가 분리되거나 애그리거트 경계를 엄격히 관리해야 하면 참조를 ID 방식으로 바꾸는 것도 고려 가능.
  - [스쿼드 세션, 다른 팀 활동 참고 정리]
    - 다른 팀들 활동을 보고 우리가 차용할 것(중요)과 참고만 할 것으로 나눠 정리함.
      - 차용하기로 한 것: 요일별로 점심메뉴 돌아가며 결정, 회의 정리 제대로 하기(데일리 스크럼만 하고 기록이 부족했음), 7팀 wiki 참고(기술적 난제 정의해두면 좋을 듯), 포인트 결제 도입 검토(2, 3, 7팀 다 넣음, 후순위지만 시도).
      - 참고용: kream 포켓몬카드 경매 UI 참고와 codex 프론트엔드 스킬로 반응형 디자인, 상태머신 유스케이스나 워크플로우로 화면 구성 구체화, API 명세서, 포인트 결제 멱등성 키, 회의 요약 남기기, Figma에 기획과 와이어프레임 정리(이후 Figma MCP로 React 마이그레이션), PWA, 기능 명세서 sheet(AI로), ERD Cloud를 DDL이나 AI 연동, 백로그는 쌓아두고 Ready에 오늘 할 것 옮기기, wiki 도입, GitHub Discussions, 영속성 컨텍스트 공부.
    - AWS 세팅 관련: 프론트를 바이브 코딩으로 진행하기로 해서, Figma 와이어프레임은 나중에 하고 일단 바이브 코딩으로 개발한 뒤 확정된 디자인 기반으로 설계하기로 함.
  - [개인 공부, POJO]
    - POJO(Plain Old Java Object)는 특정 클래스나 인터페이스를 강제 상속/구현하지 않고, 특정 프레임워크의 어노테이션이나 API에 종속되지 않으며, JDK 외 제약이 없는 순수 자바 객체. extends HttpServlet 같은 게 붙으면 POJO가 아님.
    - 지향하는 이유: 프레임워크가 바뀌어도 비즈니스 로직을 재사용할 수 있는 유연성과 재사용성, 외부 환경 없이 순수 JUnit으로 테스트 가능한 테스트 용이성, 복잡성이 낮아 OOP 원칙에 충실해지는 가독성과 유지보수성.
  - [개인 공부, ApiResponse 공통 응답 포맷]
    - 응답 구조를 통일 안 하면(성공은 그냥 DTO, 생성은 message, 에러는 timestamp/error/code 제각각) 프론트가 매번 data.error인지 data.message인지 분기해야 함.
    - ApiResponse<T>로 isSuccess, code, message, result를 규격화하면, 프론트는 isSuccess만 보고 성공이면 result를 꺼내고 실패면 message를 Toast로 띄우면 됨. RestControllerAdvice와 조합하면 모든 에러를 한 포맷으로 감싸고, HTTP 상태 코드 외에 USER_NOT_FOUND 같은 세분화된 비즈니스 코드를 줄 수 있음. 제네릭은 클래스 단 <T>로 잡음.
  - [개인 공부, Transactional과 이벤트 리스너]
    - 주 기능(DB 저장)과 부가 기능(이메일이나 알림 발송)을 한 트랜잭션에 묶으면 강한 결합, 부가 기능 실패 시 주 로직까지 롤백, 느린 외부 I/O가 DB 커넥션을 오래 점유하는 문제가 생김. 그래서 주 로직은 이벤트만 발행(publishEvent)하고 리스너가 따로 처리하게 분리.
    - EventListener는 발행 즉시 동기 실행되고 트랜잭션을 공유해서 리스너 에러 시 같이 롤백됨(그냥 한 메서드에 짠 거랑 큰 차이 없음). TransactionalEventListener는 기본 AFTER_COMMIT이라 주 트랜잭션이 커밋을 마친 직후에만 실행됨.
    - 주의점 2가지: 첫째, AFTER_COMMIT 시점엔 트랜잭션이 끝나서 새 INSERT나 UPDATE가 반영 안 됨. 리스너에서 DB 작업하려면 Propagation.REQUIRES_NEW로 새 트랜잭션을 열어야 함. 둘째, AFTER_COMMIT이어도 여전히 동기라 이메일 3초 동안 사용자가 대기함. 진짜 비동기와 결합 분리는 Async를 함께 붙임. 이벤트 객체는 상속 없는 POJO(record)로 둠.
- (오늘의 목표 중에서) 아직 끝내지 못한 부분:
  - B 트랙(입찰, 동시성) 실제 착수는 아직. 구간별 최저 상승가 정책 테이블화, 비관적 락/원자적 처리 설계, 동시성 테스트 코드 뼈대는 다음으로 넘어감.
  - 회의 정리를 제대로 하자고 정하긴 했는데 정착시킬 구체적 포맷(어디에, 어떤 항목으로 남길지)까지는 못 정함.

### 2. 오늘 가장 의미있었던 기술적 판단
- 내가 내린 판단:
  - 엔티티 참조 방식을 JPA 객체 연관관계로 확정하고, 연관관계 단점을 규칙(전부 LAZY, 단방향 기본, 컬렉션 미매핑, 상세 fetch join, 목록 DTO Projection, Cascade 제한)으로 미리 봉쇄하기로 함.
- 그렇게 판단한 이유:
  - 단일 모놀리식이고 경매 상세나 입찰 내역에서 차량, 게시글, 입찰자를 함께 조회하는 화면이 많아서, 연관관계로 두면 JPQL이나 QueryDSL 조인이 자연스러움. 지금은 애그리거트 경계를 엄격히 나누거나 MSA로 갈 필요가 명확하지 않아 전면 ID 참조는 조회 코드만 복잡하게 만드는 오버엔지니어링이라고 봄.
- 고려했지만 선택하지 않은 대안:
  - 전면 ID 참조. 경계 분리나 결합 감소엔 유리하지만 JPA 편의를 통째로 버리고 조회할 때마다 Repository로 다시 가져와야 해서 지금 규모엔 과함. 몰라서 안 쓴 게 아니라 편익 비교로 안 택함.
- 이 판단이 코드 품질/테스트/유지보수에 미친 영향:
  - 참조 방식이 하나로 통일돼 조회 코드가 단순해지고, N+1을 API별로 필요한 연관관계를 정의하고 상세는 fetch join, 목록은 DTO Projection이라는 규칙으로 팀 컨벤션화할 수 있게 됨. 엔티티를 응답으로 직접 반환하지 않기로 해서 직렬화 중 지연 로딩 사고도 사전 차단.

### 3. 가장 막혔던 문제와 해결 과정
- 문제 상황 또는 증상:
  - 어제 Bid만 ID로 빼고 나머지는 연관관계로 두려다 그럼 결국 전면 JPA랑 뭐가 다르냐에서 막혀 참조 방식을 못 정하고 끝났었음.
- 처음 추측한 원인:
  - 애그리거트 경계를 얼마나 엄격히 그을지의 문제라, 엔티티별로 최적을 따지면 답이 나올 거라고 생각함.
- 원인을 확인하기 위해 시도한 방법:
  - 엔티티 하나씩 최적을 고르는 대신, 지금 프로젝트가 실제로 뭘 필요로 하나라는 상위 기준(모놀리식인가, 조인이 잦은가, MSA 확장이 임박했나)으로 먼저 판단해봄.
- 실제 원인:
  - 애초에 엄격한 경계 분리가 지금 필요하지 않았음. 필요 없는 이점(경계 분리, 결합 감소)을 기준에 넣고 엔티티별로 저울질하니 계속 애매했던 것.
- 해결 방법:
  - 판단 기준을 현재 프로젝트 특성(모놀리식에 잦은 조인, 확장 불명확)으로 못 박고 전면 연관관계로 확정. 대신 단점은 규칙으로 봉쇄하고, 향후 경계가 실제로 필요해지면 그때 ID로 전환하는 걸 조건부로 남겨둠.
- 다음에 비슷한 문제가 생기면 먼저 확인할 것:
  - 컨벤션 성격의 결정은 엔티티나 케이스를 하나씩 보기 전에 지금 프로젝트가 그 이점을 실제로 필요로 하는가를 먼저 확인할 것. 미래 이점은 조건부 전환 항목으로 남기고 현재 결정을 붙잡지 말 것.

### 4. 오늘 잘한 행동(개발과 관련된 것)
- 어떤 행동이었는지:
  - 역할 분배 때 각자 하고 싶은 걸 먼저 구체화해서 소통하며 완만하게 배정하고, 트랙이 물리는 지점(소프트 클로즈)은 A가 소스를 쥐고 B가 트리거만 위임하는 식으로 경계를 미리 그어둠.
- 그 접근이 도움이 된 이유:
  - 하고 싶은 걸 먼저 말하게 하니 배정에 불만이 없었고, 경계를 미리 정해두니 나중에 이건 누가 하냐로 부딪힐 지점을 개발 전에 줄여둠.
- 다음에도 반복하고 싶은 점:
  - 역할을 나눌 때 작업 목록만 쪼개지 말고, 트랙 간 의존과 호출 경계까지 같이 정해두는 것.

### 5. 오늘 아쉬웠던 행동(개발과 관련된 것)
- 비효율적이었던 접근/놓치거나 부족했던 점:
  - 다른 팀에서 배운 것 중 제일 뼈아팠던 게 논의는 잘 하는데 기록을 못 남긴다였는데, 오늘도 회의 정리를 잘하자고 정하기만 하고 정작 어떤 포맷으로 어디에 남길지는 안 정함.
- 그렇게 된 원인:
