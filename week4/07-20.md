### 2. 오늘 가장 의미있었던 기술적 판단
- 내가 내린 판단:
  - 거래 방향을 특정 유형(B2C 등)으로 제한하지 않고, 딜러/일반인 누구나 매수/매도 양쪽에 참여할 수 있는 '개방형 경매 모델'(C2C/B2C/C2B/B2B 모두 허용)을 채택하기로 한 것.
- 그렇게 판단한 이유:
  - 기존 구조에서는 판매자는 '딜러 매입가(도매)' 기준으로만 팔고, 구매자는 '딜러 판매가(소매)' 기준으로만 사야 해서, 그 사이의 마진이 전부 딜러에게 귀속됨.
  - 일반인을 경매에 참여시키면 '판매자 평가액 ~ 구매자 평가액' 사이의 잉여를 당사자끼리 나눠, 판매자는 더 비싸게 팔고 구매자는 더 싸게 살 수 있음. (예: 도매 1,050 < 판매자 평가 1,100 < 구매자 평가 1,180 < 소매 1,250 → 잉여 80)
  - 단, 이 잉여는 '판매자 평가액 + 플랫폼 총비용(수수료/진단비/탁송비) < 구매자 평가액'이 성립할 때만 실질적 이득이므로, 검수/보증(평가사 진단), 에스크로, 탁송 등 신뢰/부대 서비스를 함께 제공하는 것을 전제로 함.
  - UI/기획상 이를 '딜러 마진'이라는 대립 프레임(예: "딜러에게 팔면 200만 원 손해")이 아니라, '동급 매물 소매 시세 vs 현재 최고 입찰가' 같은 중립적 시장 정보로 보여주기로 함(딜러 이탈/업계 마찰 방지).
  - 또한 복잡한 비즈니스 룰(경매 연장, 입찰 조건 검증)을 순수 자바 객체(Vehicle 등)로 관리하면 테스트가 쉽고 프레임워크 종속성을 줄일 수 있기 때문.
- 고려했지만 선택하지 않은 대안:
  - 딜러 전용 B2C 모델(기존 헤이딜러/Kcar 방식): 판매자가 도매가, 구매자가 소매가에 묶여 마진이 전부 딜러에게 귀속되므로 서비스의 핵심 차별성이 사라져 배제.
  - 일반인 전용 C2C 모델: 레몬 마켓 리스크와 낙찰 후 잠수(노쇼) 문제가 커서, 딜러까지 포함한 개방형으로 유동성과 물량(비인기 차종, 유찰/차순위 승계 물량 등)을 확보하는 편이 낫다고 판단해 배제.
  - 즉, 특정 거래 방향으로 참여자를 제한하는 대신 모든 조합(C2C/B2C/C2B/B2B)을 허용하되, 신뢰 리스크는 평가사 진단/에스크로/보증금으로 통제하는 방향을 선택함.
- 이 판단이 코드 품질/테스트/유지보수에 미칠 영향 (아직 구현 전, 기획/설계 단계에서 기대하는 효과):
  - 도메인 클래스를 프레임워크로부터 격리하면 비즈니스 로직(예: 금액이 음수인지, 모델명이 비었는지)에 대한 단위 테스트를 순수 자바 객체로 직관적으로 작성할 수 있을 것으로 기대함.
  - 딜러와 일반인의 입찰 테이블을 분리하지 않고 단일 경매로 통합하는 방향으로 잡아, DB 설계 및 쿼리 복잡도를 낮출 수 있을 것으로 예상함.

### 3. 가장 막혔던 문제와 해결 과정
- 문제 상황 또는 증상:
  - (공부하는 과정에서) 스프링 빈(Bean) 등록과 의존성 주입(DI) 중 `@Component`, `@Bean`, `@Configuration`, `@Autowired`의 정확한 사용처와 동작 원리가 헷갈려, 앞으로 클래스를 어떻게 설계해야 할지 감이 잡히지 않음.
- 처음 추측한 원인:
  - 모든 스프링 관리 객체는 무조건 `@Autowired`를 변수 위에 선언해야 의존성이 주입되는 것으로 오해함.
- 원인을 확인하기 위해 시도한 방법:
  - 스프링 공식 문서 및 실무 권장 패턴(생성자 주입)에 대한 자료를 찾아보고, `@Service`, `@RestController` 내부를 뜯어봄.
- 실제 원인:
  - 스프링 최신 버전에서는 클래스에 생성자가 딱 1개만 존재하면 `@Autowired`를 생략해도 리플렉션을 통해 자동으로 주입(생성자 주입 방식)을 처리해준다는 점을 간과함.
  - 또한, `@Component`는 클래스 레벨에서 스프링이 직접 객체를 생성할 때, `@Bean`은 `@Configuration` 안에서 외부 라이브러리 등을 수동으로 객체화하여 리턴할 때 쓴다는 명확한 기준이 없었음.
- 해결 방법:
  - 개념을 정리하고, 앞으로 실제 구현 시에는 필드 주입(`@Autowired private ...`) 대신 `private final` + 단일 생성자를 사용하는 방식(생성자 주입)을 코딩 컨벤션으로 삼기로 정리함.
- 다음에 비슷한 문제가 생기면 먼저 확인할 것:
  - 외부 라이브러리(예: RestClient)를 빈으로 등록하는지, 내가 직접 만든 클래스인지 구분하여 `@Bean`과 `@Component`를 적재적소에 사용하고 있는지 확인할 것.

### 4. 오늘 잘한 행동(개발과 관련된 것)
- 어떤 행동이었는지:
  - Java 21의 가상 스레드(Virtual Thread), 레코드 패턴(Record Pattern), 텍스트 블록 등 최신 문법과 HTTP Interface 같은 모던한 기술 요소를 비즈니스 요구사항에 연결하여 학습한 것.
- 그 접근이 도움이 된 이유:
  - 실시간 경매 시스템은 수많은 유저가 동시에 접속해 입찰(호가) API를 호출(블로킹 작업 발생)하므로, 기존 OS 커널 스레드 풀 방식 대신 초경량 가상 스레드를 도입하면 서버 전체의 타격을 줄일 수 있다는 '기술적 타당성'을 확보하게 됨.
- 다음에도 반복하고 싶은 점:
  - 새로운 기술이나 프레임워크 기능을 배울 때, 단순히 사용법만 암기하지 않고 "우리 메인 프로젝트의 어떤 트래픽 병목이나 로직 비효율을 해결할 수 있는가?"를 질문하며 학습하는 태도.

### 5. 오늘 아쉬웠던 행동(개발과 관련된 것)
- 비효율적이었던 접근/놓치거나 부족했던 점:
  - 내가 스프링 기본기가 부족한 부분이 커서, 스프링 공부에 시간을 많이 쏟고 있다는 점이 아쉬웠다. 또한 기획 단계에서, 우리 서비스만의 핵심 가치를 갖게하고, 차별점을 구체화 하기 위해 시간을 많이 쏟게 되어, 개발에 바로 투입하지 못한 부분이 아쉽다.
- 그렇게 된 원인:
  - 스프링/JPA 기본기가 아직 부족해 개념을 이해하는 데 절대적인 시간이 필요했고, 기획 회의에서도 우리 서비스만의 차별점을 합의하는 논의가 길어져 개인 개발 시간으로 전환할 여유가 없었음.
- 다음에는 다르게 해볼 구체적인 행동:
  - 내일은 학습이나 기획 회의에 시간을 설정하고, 정해진 시간이 지나면 즉시 README 작성과 API/ERD 설계(소프트 클로즈 트리거, 입찰자 수 카운팅 로직 등) 등 산출물을 만드는 구체적인 행동으로 넘어가겠다.
