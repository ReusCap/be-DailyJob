# be-DailyJob 

> 이 회고는 작업량을 보고하기 위한 문서가 아니라, 
> 오늘 내가 어떤 판단을 했고 어떤 개발 행동을 반복하거나 바꿔야 하는지 확인하기 위한 기록입니다. 
> 각 항목은 길게 쓰기보다, 사실/판단/근거/다음 행동이 드러나게 구체적으로 작성하세요.

### 1. 오늘 완료한 것
- 오늘 실제로 끝낸 기능/수정/학습:
  - [ERD 설계] 
    - 어제까지 확정하지 못했던 ERD를 여러 논의 끝에 최종 확정함. 세 가지를 정리함. 첫째, PARTICIPATION 테이블을 제거함. 경매 참여자 수를 DB로 세는 건 성능상 부담이 크고 굳이 테이블로 둘 필요가 없어서, 일단 인메모리에서 관리하고 확장이 필요해지면 그때 다시 고려하기로 함. 둘째, AUCTION 엔티티에서 base_end_time 컬럼을 제거함. 셋째, USER role에 EVALUATOR를 추가함.
  - [ERD 설계 - Evaluator 결정] 
    - EVALUATION.evaluator_id가 USER를 참조하는데 role에 평가사가 없으면 그 유저가 평가사라는 걸 스키마에서 표현할 수 없어서, 참조는 있는데 설명이 안 되는 상태가 남았음. 우리 서비스 유저 종류가 일반/딜러/평가사 세 가지이고 평가 처리 흐름이 실제로 존재하니, 일단 role을 도입해 스키마 정합성을 맞추고 확장은 나중에 고려하기로 함. enum 값 하나 추가라 비용도 거의 없음.
  - [프론트엔드 바이브코딩] 
    - 프론트엔드 바이브코딩 강의를 수강함. 실제로 도움이 많이 됨. 강의 내용을 바탕으로 프론트를 직접 바이브코딩으로 작업해서 PR까지 올림. 로컬 환경에서 pnpm으로 빌드 테스트까지 돌려보고 정상 동작 확인함.
  - [개인 공부] 
    - 로컬 개발 환경 세팅을 코드 단위로 정리함. docker-compose로 MySQL 8.4를 race-mysql 컨테이너로 띄우고 3306 포트포워딩, DB/계정/타임존/문자셋 설정, volume으로 데이터 유지까지 정리했고, build.gradle 의존성(web-mvc, springdoc, data-jpa, mysql-connector-j, lombok, test)과 implementation/compileOnly/annotationProcessor/runtimeOnly 구분, application.yaml의 datasource와 jpa 설정(ddl-auto, show-sql, format_sql)을 함께 정리함. docker-compose와 application.yaml, build.gradle 세 파일이 어떻게 맞물리는지 흐름으로 묶음.
  - [개인 공부] 
    - Swagger(SpringDoc) 도입을 정리함. /v3/api-docs로 OpenAPI JSON이 자동 생성되고 Swagger UI에서 Try it out으로 실제 요청을 테스트하는 흐름, @Tag와 @Operation의 역할, swagger-ui path 단순화, 운영 환경에서는 profile(@Profile, springdoc.enabled=false)로 Swagger를 막는 보안 실무까지 정리함.
  - [개인 공부] 
    - jakarta 표준을 정리함. Persistence(JPA), Servlet, Validation, Transaction 네 가지 핵심 기능이 각각 무슨 역할인지와 어노테이션 예시를 묶었고, @Getter/@Entity/@NotBlank/@Component/@GetMapping 같은 자주 쓰는 어노테이션이 각각 lombok/jakarta/spring 어디서 오는지 import 경로 기준으로 정리함.
- (오늘의 목표 중에서) 아직 끝내지 못한 부분:
  - EVALUATOR role을 스키마에는 넣었지만 실제 로그인 유저는 없는 상태라, 코드에서 role 권한 분기 시 EVALUATOR 케이스를 어떻게 다룰지는 아직 정하지 못함.

### 2. 오늘 가장 의미있었던 기술적 판단
- 내가 내린 판단:
  - 경매 참여자 수를 세는 PARTICIPATION 테이블을 제거하고 인메모리에서 관리하기로 함.
- 그렇게 판단한 이유:
  - 참여자 수는 실시간으로 자주 바뀌는 값이라 매번 DB에 기록하고 count 쿼리로 세면 DB 부담과 쿼리 비용이 큼. 지금 규모에서는 인메모리로 관리하는 게 성능상 훨씬 낫다고 판단함.
- 고려했지만 선택하지 않은 대안:
  - 참여자를 PARTICIPATION 테이블에 그대로 저장하고 count로 집계하는 방식. 정합성은 확실하지만 실시간 경매방 특성상 오버스펙이라 배제함. 대신 나중에 다중 서버로 확장하거나 참여 이력이 필요해지면 그때 다시 도입하기로 여지를 남겨둠.
- 이 판단이 코드 품질/테스트/유지보수에 미친 영향:
  - 엔티티가 하나 줄어 ERD가 단순해졌고, 경매방 참여자 수 로직을 DB가 아니라 애플리케이션 메모리에서 다루게 되면서 실시간 처리와 분리해 설계할 수 있게 됨. 다만 서버가 여러 대로 늘어나면 인메모리 상태 공유 문제가 생기니 확장 시점에 반드시 다시 봐야 함.

### 3. 가장 막혔던 문제와 해결 과정
- 문제 상황 또는 증상:
  - USER role에 EVALUATOR를 넣을지 말지 계속 고민함. 실제로 평가사로 로그인하는 유저는 이번 구현 범위에 없어서, 넣으면 한 번도 안 쓰이는 죽은 값이 될 수 있었음.
- 처음 추측한 원인:
  - 안 쓰는 값을 스키마에 두면 코드에서 role 분기 처리할 때 EVALUATOR 케이스를 형식적으로 넣을지 말지 애매해지니, 실제 구현 범위(일반/딜러만 로그인)에 딱 맞춰 빼는 게 깔끔하지 않을까 생각함.
- 원인을 확인하기 위해 시도한 방법:
  - role을 뺐을 때 스키마가 어떻게 되는지 되짚어봄. EVALUATION.evaluator_id가 USER를 참조하는데 그 USER가 어떤 role인지 스키마 안에서 설명이 안 되는 참조가 남았음. 넣는 쪽과 안 넣는 쪽의 장단점을 나눠 비교함.
- 실제 원인:
  - 구현 범위(안 쓰는 값 배제)와 스키마 정합성(참조를 설명할 수 있어야 함) 두 기준이 충돌한 것. 어느 한쪽만 보고 판단하려 해서 막혔음.
- 해결 방법:
  - 스키마 정합성을 우선해서 EVALUATOR를 추가하기로 함. 우리 서비스 유저 종류가 세 가지이고 평가 처리 흐름이 실제로 존재하니, 그 유저의 플로우를 스키마에 담는 게 맞다고 판단함. enum 값 하나 추가라 비용도 거의 없음.
- 다음에 비슷한 문제가 생기면 먼저 확인할 것:
  - 가정을 세워 설계할 때는 그 가정에 필요한 요소를 스키마에 일관되게 반영할 것. 구현 범위만 보고 빼면 참조는 있는데 설명이 안 되는 상태가 남을 수 있음.

### 4. 오늘 잘한 행동(개발과 관련된 것)
- 어떤 행동이었는지:
  - EVALUATOR를 넣을지 결정할 때 넣는 쪽과 안 넣는 쪽의 장단점을 나눠 적어놓고 비교한 뒤에 판단한 것. 감이 아니라 기준을 정리하고 골랐음.
- 그 접근이 도움이 된 이유:
  - 나중에 왜 EVALUATOR를 넣었냐는 질문을 받아도, 스키마 정합성 때문에 넣었고 안 넣었을 때 어떤 문제가 있었는지 근거를 바로 댈 수 있게 됨. ERD를 제출물로 볼 때 설명 안 되는 참조가 남는 지적을 피할 수 있음.
- 다음에도 반복하고 싶은 점:
  - 넣을지 뺄지 애매한 설계 결정은 장단점을 양쪽으로 적어놓고 판단하는 습관.

### 5. 오늘 아쉬웠던 행동(개발과 관련된 것)
- 비효율적이었던 접근/놓치거나 부족했던 점:
  - EVALUATOR role을 스키마에는 넣었지만, 정작 코드에서 role 권한 분기를 할 때 EVALUATOR 케이스를 어떻게 처리할지는 결정을 미룸. 넣기로 한 이유가 정합성인데 실제 코드에서 어떻게 다룰지는 안 정해서 반쪽짜리 결정이 됨.
- 그렇게 된 원인:
  - ERD 확정에 집중하느라 그 결정이 실제 코드 분기까지 어떻게 이어지는지는 뒤로 미룸.
- 다음에는 다르게 해볼 구체적인 행동:
  - 스키마에 값을 추가하기로 정할 때, 그 값을 코드에서 어떻게 다룰지(권한 분기에 포함할지, 방어 코드를 둘지)까지 같이 메모해두기.
