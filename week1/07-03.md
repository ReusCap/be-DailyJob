### 2. 오늘 가장 의미있었던 기술적 판단

- 내가 내린 판단:
  클라이언트 연결마다 Thread를 직접 생성하지 않고, ExecutorService의 고정 스레드 풀을 사용하기로 했습니다.
- 그렇게 판단한 이유:
  요청이 많아질 때 스레드가 무한정 늘어나는 것을 막고, 스레드를 재사용해 서버 자원을 안정적으로 관리하기 위해서입니다.
- 고려했지만 선택하지 않은 대안:
  기존처럼 요청마다 Thread를 새로 생성하는 방식은 단순하지만 확장성이 떨어진다고 판단했습니다.
- 이 판단이 코드 품질/테스트/유지보수에 미친 영향:
  요청 처리 구조가 더 안정적이 되었고, 로그를 통해 스레드가 재사용되는 것도 확인할 수 있었습니다.

### 3. 가장 막혔던 문제와 해결 과정

- 문제 상황 또는 증상:
  Gradle 9.3.0 환경에서 JUnit 테스트 실행 설정이 바로 되지 않았습니다.
- 처음 추측한 원인:
  JUnit 의존성 설정이 부족하거나 Gradle 버전 차이 때문이라고 생각했습니다.
- 원인을 확인하기 위해 시도한 방법:
  에러 로그를 확인하고, Gradle 9.x에서 필요한 테스트 실행 의존성을 찾아봤습니다.
- 실제 원인:
  JUnit Platform launcher 의존성을 명시적으로 추가해야 했습니다.
- 해결 방법:
  `junit-platform-launcher` 의존성을 추가해 테스트가 정상 실행되도록 수정했습니다.
- 다음에 비슷한 문제가 생기면 먼저 확인할 것:
  Gradle 버전 변경 시 테스트 런처나 플러그인 의존성 변경사항을 먼저 확인하겠습니다.

### 4. 오늘 잘한 행동

- 어떤 행동이었는지:
  기능 구현만 하고 끝내지 않고, Request Line 파싱 로직을 유틸 클래스로 분리하고 단위 테스트까지 작성했습니다.
- 그 접근이 도움이 된 이유:
  작은 로직이라도 테스트로 검증하니 코드의 책임이 더 명확해졌고, 이후 수정하기 쉬운 구조가 되었습니다.
- 다음에도 반복하고 싶은 점:
  기능을 구현할 때 바로 테스트 가능한 단위로 분리하는 습관을 계속 가져가고 싶습니다.

### 5. 오늘 아쉬웠던 행동
- 비효율적이었던 접근/놓치거나 부족했던 점:
  RequestHandler.run()에 요청 읽기, path 추출, 정적 파일 읽기, 응답 생성까지 많은 책임이 몰려 있었는데, 이를 충분히 분리하지 못했습니다. 또한 정적 파일 경로 검증, 404 응답, Content-Type 처리 같은 예외 상황도 함께 고려하지 못했습니다.
- 그렇게 된 원인:
  우선 기능이 동작하게 만드는 데 집중하다 보니, 코드의 책임 분리와 실제 HTTP 서버에서 필요한 안정성까지 깊게 보지 못했습니다.
- 다음에는 다르게 해볼 구체적인 행동:
  run()은 전체 흐름을 조립하는 역할만 하도록 줄이고, 요청 파싱, 정적 파일 조회, 응답 생성, 에러 응답 처리를 별도 메서드나 클래스로 분리하겠습니다. 기능 구현 후에는 “이 메서드가 너무 많은 일을 하고 있지 않은가?”를 먼저 점검하겠습니다.
