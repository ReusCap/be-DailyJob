# be-DailyJob

## 1. 오늘 완료한 것

- 오늘 실제로 끝낸 기능/수정/학습:

  - HTTP POST 요청은 데이터를 URL 쿼리스트링이 아니라 Request Body에 담아 전달한다는 점을 학습했다.
  - 회원가입 기능을 GET 방식에서 POST 방식으로 바꾸기 위해 Request Body를 읽고 파라미터를 추출해야 한다는 흐름을 이해했다.
  - 가입 완료 후 `/index.html`로 이동시키기 위해 HTTP 302 Redirect 응답이 필요하다는 점을 학습했다.
  - 응답 생성을 `HttpResponse`로 분리하고, 상태 코드는 `HttpStatus` enum으로 관리하는 방향을 잡았다.
  - 정적 파일 응답, 회원가입 처리, 응답 쓰기 책임을 각각 `StaticFileHandler`, `UserHandler`, `HttpResponseWriter`로 나누는 구조를 설계했다.

- 아직 끝내지 못한 부분:

  - POST 요청의 Body를 실제로 파싱해서 회원가입 기능에 완전히 연결하는 작업은 아직 마무리하지 못했다.
  - `RequestHandler`에 남아 있는 큰 책임을 각 Handler와 Writer로 완전히 분리하지 못했다.

## 2. 오늘 가장 의미있었던 기술적 판단

- 내가 내린 판단:

  - HTTP 응답 생성 로직을 `RequestHandler`에 계속 두지 않고 `HttpResponse`, `HttpStatus`, `HttpResponseWriter`로 분리하기로 했다.

- 그렇게 판단한 이유:

  - 200, 302, 404, 503 등 응답 상태가 늘어날수록 `RequestHandler` 안에서 직접 문자열을 만들면 코드가 복잡해진다고 판단했다.
  - 정적 파일 응답, 회원가입 처리, redirect 응답은 역할이 다르기 때문에 클래스를 나누는 편이 유지보수에 더 좋다고 생각했다.

- 고려했지만 선택하지 않은 대안:

  - `RequestHandler` 안에서 if문으로 모든 응답을 직접 처리하는 방식.
  - 상태 라인과 헤더를 문자열로 매번 직접 조립하는 방식.

- 이 판단이 코드 품질/테스트/유지보수에 미친 영향:

  - 응답 상태와 응답 생성 방식을 한 곳에서 관리할 수 있어 중복이 줄어들 것 같다.
  - 이후 404, redirect, static file 응답을 각각 테스트하기 쉬운 구조가 될 것 같다.

## 3. 가장 막혔던 문제와 해결 과정

- 문제 상황 또는 증상:

  - `HttpResponse`를 분리하려고 했지만, 어떤 필드와 메서드를 가져야 하는지 헷갈렸다.
  - `ok()`, `found()` 같은 메서드가 getter인지, 객체를 생성하는 정적 팩토리 메서드인지 구분이 잘 되지 않았다.
  - POST 요청에서 Body가 무엇인지, GET 요청에서 파일을 읽는 흐름과 어떻게 다른지도 혼동했다.

- 처음 추측한 원인:

  - 응답 객체를 단순히 문자열 출력용 클래스로만 생각해서 구조가 잘 잡히지 않는다고 생각했다.

- 원인을 확인하기 위해 시도한 방법:

  - `HttpStatus` enum을 만들어 상태 코드와 reason phrase를 분리해보았다.
  - `record` 기반 `HttpResponse` 예시와 기존 클래스 방식을 비교해보았다.
  - `Content-Type`, `Accept`, `requestUri`, `request body`, `response body`의 역할을 하나씩 구분해보았다.

- 실제 원인:

  - HTTP 요청과 응답의 역할을 정확히 분리하지 못한 상태에서 클래스를 먼저 나누려고 해서 헷갈렸다.
  - 특히 요청 Body와 응답 Body를 같은 개념처럼 생각한 것이 혼란의 원인이었다.

- 해결 방법:

  - `HttpResponse`는 응답 상태, 헤더, 응답 Body를 담는 객체로 보고, `ok()`, `found()`는 `HttpResponse` 객체를 생성하는 정적 팩토리 메서드로 이해했다.
  - 정적 파일 처리는 `StaticFileHandler`, 회원가입 처리는 `UserHandler`, 실제 응답 출력은 `HttpResponseWriter`로 나누는 방향으로 정리했다.

- 다음에 비슷한 문제가 생기면 먼저 확인할 것:

  - 이 클래스가 요청을 처리하는 책임인지, 응답을 만드는 책임인지, 응답을 출력하는 책임인지 먼저 구분한다.
  - 메서드를 만들기 전에 입력값, 반환값, 사용하는 위치를 먼저 정리한다.

## 4. 오늘 잘한 행동

- 어떤 행동이었는지:

  - 코드를 바로 고치기보다, `HttpStatus`, `HttpResponse`, Handler 분리 방향을 먼저 고민했다.
  - 헷갈리는 개념을 질문하면서 하나씩 확인했다.

- 그 접근이 도움이 된 이유:

  - 단순히 동작만 맞추는 것이 아니라, 앞으로 기능이 늘어났을 때 유지보수하기 좋은 구조를 생각할 수 있었다.
  - `GET`, `POST`, `request body`, `response body`, `redirect`의 차이를 더 명확히 이해하게 되었다.

- 다음에도 반복하고 싶은 점:

  - 기능 구현 전에 책임 분리를 먼저 고민하는 습관을 유지하고 싶다.
  - 모르는 개념은 추측으로 넘기지 않고, 코드 흐름과 HTTP 구조를 연결해서 확인하고 싶다.

## 5. 오늘 아쉬웠던 행동

- 비효율적이었던 접근/놓치거나 부족했던 점:

  - 처음부터 구조를 크게 바꾸려고 하다 보니 실제 POST 회원가입 기능 구현 속도가 느려졌다.
  - `ok()` 메서드, getter, 반환 타입, 파라미터의 의미를 구분하는 데 시간이 오래 걸렸다.

- 그렇게 된 원인:

  - HTTP 응답 객체를 만들어본 경험이 부족했고, record나 정적 팩토리 메서드 같은 문법도 익숙하지 않았다.
  - 요청과 응답의 데이터 흐름을 머릿속에서 명확히 나누지 못했다.

- 다음에는 다르게 해볼 구체적인 행동:

  - 먼저 최소 기능이 동작하도록 구현한 뒤, 중복되거나 책임이 큰 부분을 리팩터링한다.
  - 클래스를 분리할 때는 “이 클래스가 맡는 책임 한 문장”을 먼저 적고 구현한다.
  - 메서드를 만들 때는 반환 타입과 파라미터가 무엇을 의미하는지 먼저 주석으로 정리한 뒤 작성한다.
