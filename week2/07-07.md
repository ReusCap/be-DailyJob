# be-DailyJob

> 이 회고는 작업량을 보고하기 위한 문서가 아니라,
> 오늘 내가 어떤 판단을 했고 어떤 개발 행동을 반복하거나 바꿔야 하는지 확인하기 위한 기록입니다.
> 각 항목은 길게 쓰기보다, 사실/판단/근거/다음 행동이 드러나게 구체적으로 작성하세요.

### 1. 오늘 완료한 것

- 오늘 실제로 끝낸 기능/수정/학습:

  - 회원가입 form 요청 방식을 GET에서 POST로 변경하였다.
  - HttpRequest에서 POST body를 읽고 `application/x-www-form-urlencoded` 형식의 파라미터를 파싱하도록 구현하였다.
  - 회원가입 요청 경로를 `/user/create`로 변경하고, POST 요청 시 User를 생성해 저장하도록 구현하였다.
  - GET `/user/create` 요청에는 405 Method Not Allowed와 `Allow: POST` 헤더를 응답하도록 처리하였다.
  - 회원가입 성공 시 302 Found와 `Location: /index.html` 헤더를 응답해 리다이렉트되도록 구현하였다.
  - HttpResponse를 record로 전환하고, 200, 302, 405 응답 생성을 위한 정적 팩토리 메서드를 추가하였다.
  - 응답 전송 책임을 HttpResponseWriter로 분리하였다.
  - 정적 파일 처리 로직은 StaticFileHandler로, 회원가입 처리 로직은 UserHandler로 분리하였다.
  - HTTP 관련 클래스는 http 패키지로, 요청 처리 핸들러는 handler 패키지로 이동하였다.
  - ContentType에서 MIME 타입과 charset을 분리하고 `toHeaderValue()`에서 헤더 값을 조합하도록 수정하였다.
  - POST body 파싱, GET 회원가입 실패, POST 회원가입 리다이렉트 테스트를 추가하였다.

- (오늘의 목표 중에서) 아직 끝내지 못한 부분:

  - Service 계층은 아직 추가하지 않았다.
  - 현재 회원가입 로직이 단순 저장 수준이라, 비밀번호 정책이나 중복 아이디 검증 같은 비즈니스 규칙이 생긴 뒤 분리하는 것이 적절하다고 판단하였다.

### 2. 오늘 가장 의미있었던 기술적 판단

- 내가 내린 판단:

  - RequestHandler가 모든 요청 처리와 응답 생성을 직접 담당하지 않도록 역할을 분리하였다.
  - handler는 HttpResponse 객체만 반환하고, 실제 응답 전송은 HttpResponseWriter가 담당하도록 구조를 변경하였다.

- 그렇게 판단한 이유:

  - 기존 RequestHandler는 요청 읽기, 경로 판단, 정적 파일 처리, 회원가입 처리, 응답 전송까지 너무 많은 책임을 가지고 있었다.
  - 이 상태로 기능이 늘어나면 RequestHandler가 계속 커지고, 수정할 때 영향을 받는 범위도 커질 것이라고 판단하였다.

- 고려했지만 선택하지 않은 대안:

  - RequestHandler 안에서 if문을 추가하며 계속 기능을 확장하는 방식도 가능했다.
  - Service 계층을 바로 추가하는 방법도 고민했지만, 현재는 비즈니스 로직이 많지 않아 단순 위임 구조만 생길 것 같아 선택하지 않았다.

- 이 판단이 코드 품질/테스트/유지보수에 미친 영향:

  - 요청 처리, 응답 생성, 응답 전송의 책임이 분리되어 코드의 역할이 더 명확해졌다.
  - StaticFileHandler와 UserHandler를 분리하면서 기능별 테스트와 수정이 쉬워졌다.
  - 새로운 요청 경로가 추가되더라도 기존 코드 전체를 건드리지 않고 handler 단위로 확장할 수 있는 구조에 가까워졌다.

### 3. 가장 막혔던 문제와 해결 과정

- 문제 상황 또는 증상:

  - 회원가입 요청을 GET에서 POST로 바꾸면서, 기존처럼 query string에서만 파라미터를 읽으면 userId, password 등의 값을 가져올 수 없었다.

- 처음 추측한 원인:

  - POST 요청에서는 파라미터가 URL 뒤의 query string이 아니라 body에 담기기 때문이라고 생각하였다.

- 원인을 확인하기 위해 시도한 방법:

  - HTTP 요청 구조를 다시 확인하였다.
  - GET은 request line의 query string에 데이터가 담기고, POST는 header 이후 빈 줄 다음 body에 데이터가 담긴다는 흐름을 기준으로 코드를 점검하였다.
  - Content-Length 값을 기준으로 body를 읽어야 한다는 점을 확인하였다.

- 실제 원인:

  - HttpRequest가 기존 GET 방식 중심으로 구현되어 있어 POST body를 읽고 파라미터로 변환하는 로직이 없었다.

- 해결 방법:

  - HttpRequest에서 Content-Length를 기준으로 body를 읽도록 구현하였다.
  - Content-Type이 `application/x-www-form-urlencoded`인 경우 body를 `key=value&key=value` 형식으로 파싱해 파라미터로 저장하였다.
  - Content-Type 뒤에 `charset=UTF-8` 같은 정보가 붙을 수 있으므로 완전 일치 비교 대신 `startsWith`로 판단하였다.

- 다음에 비슷한 문제가 생기면 먼저 확인할 것:

  - 요청 메서드가 GET인지 POST인지 먼저 확인한다.
  - 파라미터가 query string에 있는지 body에 있는지 확인한다.
  - body를 읽어야 한다면 Content-Length와 Content-Type을 먼저 확인한다.

### 4. 오늘 잘한 행동(개발과 관련된 것)

- 어떤 행동이었는지:

  - 기능을 구현한 뒤, RequestHandler의 책임이 너무 커졌다는 문제를 인식하고 구조를 다시 정리하였다.
  - 패키지를 단순히 폴더처럼 나누지 않고, 같은 이유로 함께 바뀌는 코드끼리 묶는 기준으로 재구성하였다.

- 그 접근이 도움이 된 이유:

  - http 패키지에는 HTTP 요청/응답을 표현하는 코드가 모이고, handler 패키지에는 실제 요청 처리 코드가 모여 구조가 더 이해하기 쉬워졌다.
  - HttpResponseWriter를 분리하면서 handler가 응답 전송 방식까지 알 필요가 없어졌다.
  - 코드의 책임이 분명해져 이후 기능 추가나 테스트 작성이 더 쉬워졌다.

- 다음에도 반복하고 싶은 점:

  - 기능 구현이 끝난 뒤 바로 넘어가지 않고, 현재 구조에서 책임이 과하게 몰린 부분이 있는지 점검하는 습관을 유지하고 싶다.
  - 패키지를 나눌 때 “같은 이유로 변경되는가?”를 기준으로 판단하고 싶다.

### 5. 오늘 아쉬웠던 행동(개발과 관련된 것)

- 비효율적이었던 접근/놓치거나 부족했던 점:

  - 처음부터 POST 요청의 body 파싱과 Content-Type 처리 방식을 충분히 고려하지 못했다.
  - RequestHandler에 기능을 먼저 추가한 뒤에야 책임이 너무 커졌다는 문제를 인식하였다.

- 그렇게 된 원인:

  - GET 방식의 흐름에 익숙해서 POST 요청 구조를 처음부터 명확하게 반영하지 못했다.
  - 기능 동작을 먼저 맞추는 데 집중하다 보니, 역할 분리 기준을 뒤늦게 고민하였다.

- 다음에는 다르게 해볼 구체적인 행동:

  - 새로운 HTTP 메서드를 처리하기 전, request line, header, body 중 어디에서 데이터를 읽어야 하는지 먼저 정리한다.
  - 기능을 추가하기 전에 이 로직이 기존 클래스의 책임에 맞는지 먼저 확인한다.
  - RequestHandler처럼 중심이 되는 클래스는 변경할 때마다 책임이 커지고 있지 않은지 계속 점검한다.
