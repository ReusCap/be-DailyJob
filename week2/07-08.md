# be-DailyJob

> 이 회고는 작업량을 보고하기 위한 문서가 아니라,
> 오늘 내가 어떤 판단을 했고 어떤 개발 행동을 반복하거나 바꿔야 하는지 확인하기 위한 기록입니다.
> 각 항목은 길게 쓰기보다, 사실/판단/근거/다음 행동이 드러나게 구체적으로 작성하세요.

### 1. 오늘 완료한 것

- 오늘 실제로 끝낸 기능/수정/학습:

  - Step4까지의 구현 흐름을 정리했다.
  - 현재 서버 흐름을 `WebServer → RequestHandler → HttpRequest 파싱 → Handler 선택 → HttpResponse 생성 → Writer 출력`으로 이해했다.
  - 현재 프로젝트가 Spring과 Tomcat이 내부에서 해주는 요청 처리 과정을 직접 구현하는 프로젝트라는 점을 정리했다.
  - 쿠키/세션의 동작 방식과 `Set-Cookie`, `Cookie` 헤더의 흐름을 학습했다.
  - Router, Route, Handler 구조와 함수형 인터페이스 개념을 정리했다.

- (오늘의 목표 중에서) 아직 끝내지 못한 부분:

  - 세션 저장소, 로그아웃, 만료 처리 로직을 실제 코드로 완전히 반영하지는 못했다.
  - Router 구조를 설계는 했지만 전체 요청 처리 흐름에 완전히 통합하는 작업은 남아 있다.

### 2. 오늘 가장 의미있었던 기술적 판단

- 내가 내린 판단:

  - Handler마다 다른 시그니처를 유지하지 않고, `HttpResponse handle(HttpRequest request)` 형태의 공통 인터페이스로 통일하기로 했다.

- 그렇게 판단한 이유:

  - StaticFileHandler와 UserHandler의 호출 방식이 다르면 Router가 핸들러를 일관되게 다루기 어렵기 때문이다.
  - 인터페이스로 계약을 정해두면 새로운 핸들러가 추가되어도 같은 방식으로 처리할 수 있다.

- 고려했지만 선택하지 않은 대안:

  - 기존 핸들러를 유지하고 등록 시점에 람다로 감싸는 방식도 고려했다.
  - 하지만 로그인, 로그아웃 등 핸들러가 늘어나면 등록 코드가 복잡해질 수 있어 선택하지 않았다.

- 이 판단이 코드 품질/테스트/유지보수에 미친 영향:

  - Handler의 책임과 형태가 명확해져 코드 구조가 단순해진다.
  - 새 기능을 추가할 때 따라야 할 규칙이 코드로 강제되어 유지보수성이 좋아진다.
  - 요청을 넣었을 때 어떤 Handler가 실행되는지 테스트하기 쉬워진다.

### 3. 가장 막혔던 문제와 해결 과정

- 문제 상황 또는 증상:

  - StaticFileHandler와 UserHandler의 입력값이 달라 Router가 공통 방식으로 처리하기 어려웠다.

- 처음 추측한 원인:

  - 단순히 Handler 이름이나 위치를 잘못 잡은 문제라고 생각했다.

- 원인을 확인하기 위해 시도한 방법:

  - 현재 요청 처리 흐름을 다시 정리했다.
  - Spring MVC와 Tomcat이 내부에서 어떤 역할을 나눠 처리하는지 비교했다.
  - 계층형 아키텍처, 도메인형 아키텍처, Router/Route 구조를 함께 검토했다.

- 실제 원인:

  - Handler의 책임과 인터페이스가 통일되어 있지 않아 RequestHandler와 Router가 너무 많은 판단을 해야 하는 구조였다.

- 해결 방법:

  - `Handler` 함수형 인터페이스를 만들고, 모든 Handler가 `handle(HttpRequest request)`를 구현하도록 설계했다.
  - `Route`는 요청이 자신과 매칭되는지만 판단하고, `Router`는 매칭된 Handler에게 요청을 위임하도록 역할을 나눴다.

- 다음에 비슷한 문제가 생기면 먼저 확인할 것:

  - 객체마다 호출 방식이 다른지 먼저 확인한다.
  - 특정 클래스가 너무 많은 분기와 책임을 가지고 있지 않은지 먼저 본다.
  - 공통 인터페이스로 추상화할 수 있는지 검토한다.

### 4. 오늘 잘한 행동(개발과 관련된 것)

- 어떤 행동이었는지:

  - 단순히 코드를 바로 작성하지 않고, 현재 프로젝트가 어떤 구조를 직접 구현하는 것인지 먼저 정리했다.
  - 쿠키/세션, Router, Handler, 함수형 인터페이스 개념을 코드 구조와 연결해서 학습했다.

- 그 접근이 도움이 된 이유:

  - 지금 만드는 코드가 Spring과 Tomcat의 어떤 역할에 해당하는지 이해할 수 있었다.
  - 단순 구현이 아니라 요청 처리 흐름 전체를 기준으로 구조를 판단할 수 있었다.

- 다음에도 반복하고 싶은 점:

  - 기능을 추가하기 전에 먼저 요청 흐름과 책임 분리를 정리한다.
  - 새 개념을 배울 때 실제 프로젝트 코드에서 어디에 쓰이는지 연결해서 이해한다.

### 5. 오늘 아쉬웠던 행동(개발과 관련된 것)

- 비효율적이었던 접근/놓치거나 부족했던 점:

  - 쿠키/세션, 아키텍처, 디자인 패턴, 함수형 인터페이스까지 한 번에 많이 정리하다 보니 실제 코드 반영 속도는 느렸다.
  - 세션 저장소와 로그아웃 처리처럼 실제 구현해야 할 부분을 아직 끝내지 못했다.

- 그렇게 된 원인:

  - 개념을 충분히 이해하고 넘어가려다 보니 학습 범위가 커졌다.
  - 구현 단위를 더 작게 쪼개지 못했다.

- 다음에는 다르게 해볼 구체적인 행동:

  - 먼저 `Router + Handler 인터페이스 통일`처럼 작은 단위 하나를 끝낸 뒤 다음 개념으로 넘어간다.
  - 세션은 `생성 → 조회 → 삭제 → 만료 처리` 순서로 나눠서 구현한다.
  - 학습한 개념은 바로 작은 테스트 코드나 예제 코드로 확인한다.
