# be-DailyJob

> 이 회고는 작업량을 보고하기 위한 문서가 아니라,
> 오늘 내가 어떤 판단을 했고 어떤 개발 행동을 반복하거나 바꿔야 하는지 확인하기 위한 기록입니다.
> 각 항목은 길게 쓰기보다, 사실/판단/근거/다음 행동이 드러나게 구체적으로 작성하세요.

### 1. 오늘 완료한 것

- 오늘 실제로 끝낸 기능/수정/학습:
  로그인 form의 요청 경로와 방식을 `/user/login`, POST로 수정하고, 서버에서 파싱하는 파라미터명과 input name을 일치시켰다.
  로그인 성공 시 세션 ID를 발급하고 `SessionStorage`에 세션을 저장한 뒤, `Set-Cookie`로 `sid` 쿠키를 내려주도록 구현했다.
  세션 저장소는 멀티스레드 요청 환경을 고려해 `ConcurrentHashMap` 기반으로 구성했고, 세션 조회, 로그인 사용자 조회, 세션 삭제 기능을 구현했다.
  쿠키 생성 책임은 `Cookie` 클래스로 분리하고, 요청 쿠키 조회는 `HttpRequest.getCookie()`에서 처리하도록 구현했다.
  로그인 실패 시 로그인 페이지로 리다이렉트하고, 로그아웃 시 서버 세션 삭제와 브라우저 쿠키 만료를 함께 처리하도록 구현했다.
  `/user/login`, `/user/logout` 요청을 각각 `LoginHandler`, `LogoutHandler`로 위임하도록 라우팅을 추가했다.

- (오늘의 목표 중에서) 아직 끝내지 못한 부분:
  로그인 실패 사유를 화면에 표시하는 처리, 세션 만료 시간 설정, 만료 세션 정리 스케줄러, 로그인 상태에 따른 동적 HTML 변경, 세션 저장소 단위 테스트는 아직 구현하지 못했다.
  또한 `RequestHandler`의 if-else 분기가 계속 늘어나고 있어, 공통 `Handler` 인터페이스와 `List<Route>` 기반 라우팅 구조로 개선하는 작업도 다음 단계로 남겨두었다.

### 2. 오늘 가장 의미있었던 기술적 판단

- 내가 내린 판단:
  로그인 세션 저장소를 단순 `HashMap`이 아니라 `ConcurrentHashMap<String, Session>`으로 구현하고, 세션 값도 `User`가 아니라 `Session` 객체로 감싸서 저장하기로 판단했다.

- 그렇게 판단한 이유:
  현재 서버는 요청마다 별도의 스레드에서 처리될 수 있으므로, 세션 저장소는 여러 요청 스레드가 동시에 접근하는 공유 상태라고 봤다.
  따라서 일반 `HashMap`을 사용하면 동시 수정 상황에서 데이터 정합성이 깨질 수 있다고 판단했다.
  또한 `sessionId -> User` 구조는 단순하지만, 이후 로그인 시각, 만료 시간, 갱신 시간 같은 세션 메타데이터를 추가하기 어렵다고 봤다.

- 고려했지만 선택하지 않은 대안:
  처음에는 `HashMap<String, User>`처럼 단순하게 세션 ID와 사용자를 바로 매핑하는 방식도 고려했다.
  하지만 로그인 세션은 서버 전체에서 공유되는 상태이고, 추후 세션 만료 같은 기능이 들어갈 가능성이 높기 때문에 확장성과 안정성을 고려해 선택하지 않았다.

- 이 판단이 코드 품질/테스트/유지보수에 미친 영향:
  세션 관련 정보가 `Session` 모델로 분리되면서 세션 저장소의 역할이 더 명확해졌다.
  이후 세션 만료 시간이나 갱신 시간 필드를 추가하더라도 저장소 구조를 크게 바꾸지 않아도 된다.
  또한 동시성 문제를 미리 고려한 자료구조를 사용하면서 로그인 기능의 안정성을 높일 수 있었다.

### 3. 가장 막혔던 문제와 해결 과정

- 문제 상황 또는 증상:
  로그아웃을 구현할 때 브라우저 쿠키만 삭제하면 충분한지, 서버 세션 저장소에서도 세션을 삭제해야 하는지 고민이 있었다.

- 처음 추측한 원인:
  클라이언트가 더 이상 `sid` 쿠키를 보내지 않으면 로그인 상태가 끊기므로, 브라우저 쿠키 만료만으로도 로그아웃이 처리될 수 있다고 생각했다.

- 원인을 확인하기 위해 시도한 방법:
  세션 쿠키의 역할과 서버 세션 저장소의 역할을 분리해서 생각했다.
  로그인 성공 시 서버에는 세션 ID와 세션 정보가 저장되고, 브라우저는 그 세션 ID만 쿠키로 보관한다는 흐름을 다시 정리했다.
  또한 curl과 개발자 도구를 사용해 로그인 시 `Set-Cookie: sid=...`가 내려오는지, 로그아웃 시 `Max-Age=0`이 포함되는지 확인했다.

- 실제 원인:
  쿠키는 클라이언트가 가진 세션 ID일 뿐이고, 실제 로그인 상태의 유효성은 서버 세션 저장소에 남아 있는 세션 정보에 의해 결정된다.
  따라서 브라우저 쿠키만 만료하고 서버 세션을 삭제하지 않으면, 같은 세션 ID가 다시 전달될 경우 서버에서는 여전히 유효한 세션으로 판단할 수 있다.

- 해결 방법:
  로그아웃 시 요청 쿠키에서 `sid` 값을 읽고, 해당 세션 ID를 기준으로 `SessionStorage.remove()`를 호출해 서버 세션을 먼저 무효화했다.
  이후 `Cookie.expired()`를 사용해 `Max-Age=0`인 쿠키를 응답으로 내려 브라우저 쿠키도 함께 만료시켰다.

- 다음에 비슷한 문제가 생기면 먼저 확인할 것:
  인증이나 상태 관리 기능을 구현할 때는 클라이언트에 저장된 값과 서버에 저장된 값의 책임을 먼저 구분해야 한다.
  특히 로그아웃, 만료, 삭제처럼 상태를 없애는 기능은 클라이언트 쪽 처리만으로 충분한지, 서버 쪽 상태도 함께 정리해야 하는지 먼저 확인해야 한다.

### 4. 오늘 잘한 행동(개발과 관련된 것)

- 어떤 행동이었는지:
  로그인, 세션, 쿠키, 로그아웃 기능을 한 번에 핸들러 내부에 몰아넣지 않고 `SessionStorage`, `Session`, `Cookie`, `LoginHandler`, `LogoutHandler`로 책임을 나누어 구현했다.

- 그 접근이 도움이 된 이유:
  각 클래스가 맡는 역할이 분명해져서 로그인 흐름을 이해하기 쉬워졌다.
  쿠키 문자열 조립은 `Cookie`가 담당하고, 요청 쿠키 조회는 `HttpRequest`가 담당하게 되면서 핸들러는 로그인과 로그아웃의 흐름에만 집중할 수 있었다.
  덕분에 이후 세션 만료, 로그인 상태 확인, 라우팅 개선 같은 기능을 추가할 때 수정 범위가 줄어들 가능성이 높아졌다.

- 다음에도 반복하고 싶은 점:
  기능을 구현하기 전에 “이 책임이 어느 객체에 있어야 자연스러운가”를 먼저 생각하는 방식을 반복하고 싶다.
  또한 브라우저 화면만으로 검증하기 어려운 기능은 curl과 개발자 도구를 함께 사용해 요청/응답 헤더까지 직접 확인하는 습관을 계속 유지하고 싶다.

### 5. 오늘 아쉬웠던 행동(개발과 관련된 것)

- 비효율적이었던 접근/놓치거나 부족했던 점:
  세션 저장소와 쿠키 처리 구조는 어느 정도 분리했지만, 아직 세션 저장소 단위 테스트를 작성하지 못했다.
  또한 `/user/login`, `/user/logout` 같은 요청 분기가 `RequestHandler`에 계속 추가되면서 라우팅 책임이 점점 커지고 있다.

- 그렇게 된 원인:
  우선 로그인과 로그아웃의 전체 동작을 완성하는 데 집중하다 보니, 테스트와 라우팅 구조 개선을 같은 PR 안에서 함께 처리하지 못했다.
  현재 단계에서는 if-else 분기로도 기능 구현이 가능했기 때문에 구조 개선을 뒤로 미뤘다.

- 다음에는 다르게 해볼 구체적인 행동:
  다음 작업에서는 세션 저장소의 `create`, `findSession`, `findUser`, `remove` 동작을 단위 테스트로 먼저 보강할 것이다.
  이후 핸들러가 더 늘어나기 전에 공통 `Handler` 인터페이스와 `List<Route>` 기반 라우팅 구조를 도입해 `RequestHandler`의 책임을 줄일 것이다.
  기능 구현이 끝난 뒤에는 “동작은 되지만 구조적으로 커지고 있는 부분이 있는지”를 체크리스트처럼 확인하는 습관을 들이고 싶다.
