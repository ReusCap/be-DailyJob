# be-DailyJob

> 이 회고는 작업량을 보고하기 위한 문서가 아니라,
> 오늘 내가 어떤 판단을 했고 어떤 개발 행동을 반복하거나 바꿔야 하는지 확인하기 위한 기록입니다.
> 각 항목은 길게 쓰기보다, 사실/판단/근거/다음 행동이 드러나게 구체적으로 작성하세요.

### 1. 오늘 완료한 것
- 오늘 실제로 끝낸 기능/수정/학습:
  - Router 도입 — RequestHandler 안에 있던 경로 분기(if-else 성격)를 `Map<String, Handler>` 기반 라우팅으로 분리. `routes.getOrDefault(path, staticFileHandler)`로 "등록된 경로면 해당 핸들러, 아니면 정적 파일" 흐름을 한 줄로 정리함.
  - `SessionManager.getLoginUser()` / `isLoggedIn()` 구현 — 요청의 쿠키(sid)로 세션을 찾아 User를 반환하거나, 로그인 여부를 boolean으로 판별하는 함수. 6단계 동적 HTML의 판단 기준이 되는 부분.
  - SessionStorage를 session 패키지로 이동 (역할 기준 패키지 정리).
  - 학습: 정적 팩토리 메서드(new를 안 쓰는 3가지 이유), 컬렉션·람다·Stream(중간/최종 연산, 원본 불변, 1회성), 메서드 참조 3종, Spring에서 DTO 변환에 Stream이 쓰이는 맥락.
- (오늘의 목표 중에서) 아직 끝내지 못한 부분:
  - 6단계 본체인 "동적 index.html" — 로그인 시 사용자 이름 표시 / 비로그인 시 [로그인] 버튼(StringBuilder)은 미착수. 오늘 만든 `isLoggedIn()`은 이걸 위한 사전 준비였음.
  - /mypage 접근 제어, 로그인 상태에서 회원가입·로그인 페이지 접근 차단(리다이렉트)도 미착수.

### 2. 오늘 가장 의미있었던 기술적 판단
- 내가 내린 판단:
  - 경로 분기를 RequestHandler에서 빼내 Router라는 별도 클래스의 Map으로 옮겼다.
- 그렇게 판단한 이유:
  - RequestHandler의 책임은 "소켓 읽기 → 요청 파싱 → 응답 쓰기"인데, 여기에 `if (path.equals("/user/create")) ... else if ...`가 계속 쌓이면 경로가 늘 때마다 이 클래스를 수정해야 한다(변경 이유가 여러 개 = SRP 위반).
  - Map으로 빼면 새 경로는 `routes.put(...)` 한 줄 추가로 끝난다. RequestHandler는 다시 안 건드림 → OCP(확장에는 열려있고 수정에는 닫혀있다)에 가까워짐.
- 고려했지만 선택하지 않은 대안:
  - enum 기반 라우팅(경로마다 enum 상수 + 핸들러). 타입 안전성은 더 좋지만 지금 경로 수(3개)에선 과설계라 판단. Map이 더 단순하고 충분함.
  - switch 문 유지: 읽기는 쉽지만 결국 분기가 코드에 박혀서 "데이터로 다루기"가 안 됨.
- 이 판단이 코드 품질/테스트/유지보수에 미친 영향:
  - 라우팅 규칙이 "코드(분기문)"에서 "데이터(Map 엔트리)"로 바뀌어서, 경로 추가가 설정처럼 바뀜. 단, 지금 Router는 HashMap인데 멀티스레드 환경(스레드풀)에서 여러 요청이 동시에 읽는다 → 생성자에서만 put하고 이후 read-only라 현재는 안전하지만, 이 "불변 전제"를 명시적으로 못 박는 게 다음 개선점(아래 5번).

### 3. 가장 막혔던 문제와 해결 과정
(오늘은 큰 디버깅 이슈보다 "구조 설계 고민"이 중심이었어. 그 지점을 문제로 기록.)
- 문제 상황 또는 증상:
  - "동적 HTML을 어디서 만들지?"가 명확하지 않았다. index.html은 정적 파일인데, 로그인 여부에 따라 내용이 달라져야 함.
- 처음 추측한 원인:
  - LoginHandler처럼 /index.html 전용 핸들러를 새로 만들까 생각.
- 원인을 확인하기 위해 시도한 방법:
  - 현재 흐름을 되짚음 — /index.html은 Router에서 매칭되는 경로가 없어 staticFileHandler로 떨어진다.
- 실제 원인:
  - 문제는 "핸들러가 없다"가 아니라 "정적 파일을 세션 상태에 따라 후처리할 지점이 없다"는 것.
- 해결 방법:
  - `SessionManager.isLoggedIn()`을 재료로, index.html을 읽은 뒤 StringBuilder로 특정 영역만 치환하는 방식(내일 실행). 전용 핸들러 신설 vs StaticFileHandler 내부 분기 중 선택 필요(4번/6번에서 다룸).
- 다음에 비슷한 문제가 생기면 먼저 확인할 것:
  - "새 클래스가 필요한 문제인가, 기존 흐름에 후처리 한 단계가 빠진 문제인가"를 먼저 구분하기.

### 4. 오늘 잘한 행동(개발과 관련된 것)
- 어떤 행동이었는지:
  - 6단계 본체(동적 HTML)를 바로 짜지 않고, 먼저 `isLoggedIn()`/`getLoginUser()`라는 판단 함수와 Router 구조부터 갖췄다.
- 그 접근이 도움이 된 이유:
  - 동적 HTML 로직 = "로그인 여부 판단" + "HTML 조립"의 두 관심사인데, 판단 부분을 SessionManager로 미리 분리해두면 내일 렌더링 코드는 `if (isLoggedIn(req))`처럼 의도가 읽히는 코드가 된다. 쿠키 파싱·세션 조회 디테일이 렌더링 코드에 안 섞임.
- 다음에도 반복하고 싶은 점:
  - 큰 기능을 "판단 / 실행"으로 쪼개서, 판단 로직을 먼저 이름 있는 함수로 분리하는 습관.

### 5. 오늘 아쉬웠던 행동(개발과 관련된 것)
- 비효율적이었던 접근/놓치거나 부족했던 점:
  - Router의 routes가 HashMap이고, "생성 후 불변"이라는 전제가 코드에 드러나 있지 않다. 지금은 안전하지만 나중에 누가 런타임에 put하면 스레드 문제가 생길 여지.
- 그렇게 된 원인:
  - "일단 동작"에 초점을 둬서, 앞서 DB/SessionStorage는 ConcurrentHashMap으로 멀티스레드를 신경 썼으면서 Router는 그 기준을 일관되게 적용 안 함.
- 다음에는 다르게 해볼 구체적인 행동:
  - 생성자에서 채운 뒤 `Map.copyOf(routes)` 또는 `Collections.unmodifiableMap`으로 감싸 불변임을 코드로 선언하기. (이건 오늘 배운 `List.of` 같은 정적 팩토리 = "불변 컬렉션 반환" 개념과 바로 연결됨.)

### 📌 내일 할 일 (6단계 완료 목표)
우선순위 순서 — 앞의 3개가 6단계 기능요구사항 핵심이야.

1. 동적 index.html 렌더링 (제일 중요, StringBuilder)
   - `SessionManager.getLoginUser(request)`로 유저 판별 → 로그인이면 사용자 이름, 아니면 [로그인] 버튼.
   - 먼저 결정할 것: index.html 요청을 ⓐ StaticFileHandler 안에서 "index면 세션 보고 치환" 분기로 넣을지, ⓑ /index.html 전용 핸들러를 Router에 등록할지. → 정적/동적 책임을 섞지 않으려면 ⓑ(전용 핸들러)가 더 깔끔하다고 추천. (StaticFileHandler는 "파일 그대로 서빙"이라는 단일 책임 유지)
2. /mypage 접근 제어
   - `isLoggedIn(request)`가 false면 `HttpResponse.found("/login/index.html")`로 리다이렉트. true면 mypage 서빙.
3. 로그인 상태에서 회원가입/로그인 페이지 접근 차단
   - 로그인 상태로 /login, /registration 접근 시 `found("/index.html")`. (이미 만든 302 리다이렉트 재사용)
4. 오늘 아쉬운 점 반영(리팩터링): Router의 routes를 불변 Map으로 감싸기 (`Map.copyOf`).
5. (여유 되면) 학습 연결: 오늘 공부한 Stream/정적 팩토리를 억지로 끼워넣지 말고, 실제 필요한 곳(예: DTO 변환, 컬렉션 반환)이 나오면 적용. 지금 단계에선 아직 강제로 쓸 곳이 많지 않음 — "배웠으니 쓴다"가 아니라 "필요해서 쓴다" 기준 유지.
